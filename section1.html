What is React Native?

Imagine you want to build an app that works on both an iPhone and an Android phone. Historically, this meant writing the app twice — once in Swift (for iOS) and once in Kotlin or Java (for Android). Two codebases, two teams, two sets of bugs, two release schedules.

React Native is a framework that lets you write your app once, using JavaScript, and have it run as a real, native app on both platforms. Not a website wrapped in an app shell — an actual native app, with real native buttons, real native scroll behaviour, and real native performance.

Why it exists Facebook (now Meta) built React Native in 2015 because their engineers already knew React (a JavaScript library for building web interfaces) and wanted a way to reuse that knowledge and a large chunk of code across web, iOS, and Android — instead of maintaining three completely separate teams and codebases.

"Native" is the key word

This is the part beginners misunderstand most, so let's be precise. There are three broad ways to build a mobile app:

1
Fully Native

Swift/Kotlin written directly for one platform. Fastest, but you write everything twice.

2
WebView Wrapper

A website stuffed inside an app shell. Easy to build, but feels and performs like a website, not an app.

3
React Native

JavaScript describes the UI, but it's rendered using real native components — a real UIView on iOS, a real View on Android.

So when you write <Text>Hello</Text> in React Native, it does not draw text on a canvas or inside a browser. It creates an actual native text label on the device. That's why RN apps feel native — because they are.

A first taste of the code

Here is the smallest possible React Native screen. Don't worry about understanding every symbol yet — we'll build up to this piece by piece over the next topics.

App.tsx 
import { Text, View } from 'react-native';

export default function App() {
  return (
    <View style={{ flex: 1, justifyContent: 'center', alignItems: 'center' }}>
      <Text>Hello, React Native! ??</Text>
    </View>
  );
}
  • import { Text, View } — we're pulling in two building blocks from the react-native library. View is like a <div> on the web: an invisible container. Text is the only component allowed to actually display text.
  • export default function App() — this defines your component. Every screen in React Native is just a JavaScript function that returns what should appear on screen.
  • style={{ flex: 1, ... }} — this makes the container fill the whole screen and center its contents. We'll cover Flexbox in full detail in Section 4.
  • If you removed the View wrapper, the Text would still render, but you'd lose the ability to center or lay out multiple elements together.

What you're looking at

This phone frame is what every code example in this course renders into. It's a simulated screen so you can see the result immediately, without installing anything yet. Once we reach "Setting Up Your Environment," you'll run this exact code on a real simulator or your own phone.

Why should you learn it?

  • One codebase, two platforms. Most business logic, and often 90%+ of UI code, is shared between iOS and Android.
  • Huge job market. Companies like Instagram, Discord, Shopify, and Coinbase ship production apps built with React Native.
  • You already half-know it if you know React. Components, props, state, hooks — all identical to web React. The main difference is which "building blocks" (components) you use.
  • Fast iteration. Changes appear almost instantly during development thanks to Fast Refresh — no waiting minutes for a full native rebuild every time.

What problem does it actually solve?

Before frameworks like this existed, a startup that wanted an iOS and an Android app needed two separate engineering teams with different skill sets, shipping features at different speeds, often with subtle behavioural differences between platforms. React Native collapses that into one team, one codebase, and one mental model — while still producing apps that look and feel fully native.

Real-world example Discord's mobile app is built with React Native. Their web team and mobile team can share logic (like message formatting or notification handling), and a single engineer familiar with React can meaningfully contribute to the mobile app without learning Swift or Kotlin from scratch.
Beginner mistakeAssuming React Native apps are "just websites in an app." They're not — there's no browser involved at runtime, and there's no HTML/CSS/DOM.
Intermediate mistakeTrying to use web-only libraries (that touch the DOM, like jQuery or many CSS frameworks) inside React Native — they simply won't work, because there is no DOM.
Professional mistakeOver-abstracting shared code between web-React and React Native too early, before understanding where the platforms genuinely diverge (navigation, gestures, native modules).

Where did the DOM go?

If you come from web development, you might be wondering why you can't use your favorite web tools like jQuery in React Native.

In a web browser, everything you see is built using HTML. The browser reads this HTML and creates a tree-like structure in its memory called the DOM (Document Object Model). Libraries like jQuery were designed specifically to search through this DOM and manipulate it.

React Native is not a web browser. It does not use HTML, and therefore, it does not have a DOM. Instead, it uses real native components (like iOS views). Because there is no HTML DOM for jQuery to interact with, it simply has nothing to do, which is why web-only libraries don't work in React Native.

How React Native Works

Now that you know what React Native is, let's open the hood and understand how your JavaScript code turns into a real native screen. You don't need to memorize this diagram, but having a mental model of it will make debugging far less confusing later.

The three worlds

A React Native app has three main "worlds" talking to each other:

1. The JavaScript World

Think of this as the "Brain" of your app. This is where you spend almost all of your time. You write normal JavaScript (or TypeScript) and React code here. It holds all your app's logic, rules, and data (like whether a user is logged in). Because mobile phones can't naturally read JavaScript, this code runs inside a special environment called Hermes (a lightning-fast JavaScript engine built specifically for React Native).

2. The Native World

Think of this as the "Body and Muscles" of your app. This world is entirely made of the raw materials that Apple (iOS) and Google (Android) provide. It includes access to the phone's camera, GPS, and the actual physical buttons drawn on the screen. The native world is incredibly fast and smooth, but it only speaks its own languages (Swift for iOS, Kotlin for Android). It doesn't understand your JavaScript at all.

3. The Bridge / JSI

Think of this as the "Nervous System" or "Translator". Since the JavaScript Brain and the Native Body speak completely different languages, they need a way to talk to each other. When your JavaScript code says, "Draw a blue button," this layer translates that command and sends it to the Native world, which then physically draws a real iOS or Android button. Modern React Native uses a super-fast translator called JSI (JavaScript Interface) that lets them talk instantly without delays.

What happens when your app starts

  1. Starting the Engine: When you tap the app icon on your phone, a standard, empty iOS or Android app opens up. Think of this as the empty frame of a car waiting for instructions.
  2. Waking up the Brain: Immediately, that empty app turns on the Hermes JavaScript Engine. It loads up your entire React Native code (which has been bundled into one big file) and brings the "JavaScript Brain" to life.
  3. Drawing the Blueprint: Your React code runs and creates a blueprint of what the screen should look like. It doesn't actually paint colors or pixels; instead, it makes a structured list of instructions like, "I need a big title at the top, and a blue button below it."
  4. Sending the Instructions: This blueprint is quickly passed over the Bridge (or JSI translator) from the JavaScript world to the Native world.
  5. Building the Real UI: The Native "Body" receives the blueprint and follows the instructions. It reaches into Apple or Google's official toolbox and constructs the actual, physical interface on your screen. You aren't getting a fake web button—you are getting a 100% real iPhone (UILabel) or Android (TextView) component.
  6. Smart Updates: When something in your app changes (like when you tap a button to 'Like' a post), React doesn't redraw the entire screen from scratch. Instead, the JavaScript "Brain" quickly figures out exactly what small piece changed (e.g., just the heart icon turning red). It then sends a tiny, fast message across the Bridge saying, "Hey, just update the color of that one heart icon." This smart process is what keeps your app running incredibly fast without draining the phone's battery.

Why this matters for you React Native still uses a bridge to send instructions from the JavaScript side to the native side. That means the app feels smooth only when the JavaScript does not send too many updates at once. If your app changes state too often — for example, updating the screen 60 times every second without care — the bridge can become a slowdown point. The app then feels less smooth because the phone spends too much time moving data back and forth. Later lessons teach safer ways to animate and update the UI, such as using useNativeDriver. These techniques keep more work on the native side and help your app stay fast.

The Old Architecture vs. the New Architecture

Old vs New Architecture — explained simply

Think of React Native's architecture like how you send messages to a friend. The old way and new way both get the job done, but they work differently.

Communication: In the old system, React Native packs many instructions into JSON messages (like writing letters) and sends them across a "bridge." This takes time because messages have to travel back and forth. The new system uses something called JSI (JavaScript Interface) that lets JavaScript talk directly to native code — like having a direct phone line instead of sending letters.

Rendering: The old system had a separate renderer that converted your JavaScript into native views. The new system uses "Fabric," which is smarter and works more like how the phone's own system works. This means your app feels more natural and responsive.

Native modules: When your app needs to use phone features like the camera or GPS, it uses "native modules." The old system loaded all of these at startup (like carrying a heavy backpack everywhere). The new system uses "TurboModules" that load only when you need them (like grabbing tools from a toolbox only when you use them).

Performance: The old bridge can get crowded when your app sends lots of updates — like traffic during rush hour. The new system has less waiting and handles busy moments better, especially for apps with lots of animations or frequent screen updates.

Old Architecture (Bridge) New Architecture (JSI + Fabric)
Communication Asynchronous, batched JSON messages Direct, synchronous-capable calls via JSI
Rendering Separate rendering pipeline Fabric renderer, tighter to native
Native modules Bridge modules TurboModules, loaded lazily
Performance Good, but bridge can bottleneck Lower overhead, better for heavy UI work

We'll revisit the New Architecture in more depth in Section 2, since Expo now defaults new projects to it.

Beginner mistakeThinking your JavaScript directly draws pixels. It doesn't — it produces a description that native code turns into real views.
Intermediate mistakeBlaming "React Native is slow" for issues actually caused by too many bridge crossings in a tight loop, rather than the architecture itself.
Professional mistakeNot profiling before optimizing — assuming you know where the bridge/JSI bottleneck is without measuring it first.

Why Expo and Expo Go

Raw React Native requires you to install and configure Xcode (for iOS) and Android Studio (for Android) before you can even see "Hello World" on a screen. That setup alone can take beginners hours and is a common reason people give up before writing a single line of app logic.

Expo is a set of tools and services built on top of React Native that removes almost all of that friction. Think of React Native as the engine, and Expo as the car built around it — steering wheel, dashboard, and all the parts that make it actually usable day-to-day.

Expo SDK

A curated library of ready-made modules — camera, location, notifications, file system — all pre-configured and tested together.

Expo Go

A free app you install on your own phone. Scan a QR code and your in-progress app runs instantly — no cables, no Xcode.

EAS

Expo Application Services — cloud build, submit, and update tooling, covered in depth in Sections 12–15.

What Expo Go actually does

Expo Go is a pre-built native app that already contains the native code for the entire Expo SDK. When you run npx expo start, your computer starts a local server serving your JS bundle. Expo Go downloads and runs that JS inside itself. This is why you can see live changes in seconds — you're not rebuilding native code every time, just re-running JavaScript inside an app that's already installed.

Where Expo Go stops being enough

Think of Expo Go like a pre-furnished apartment. It comes with almost everything you need to start building immediately (like camera access, GPS, and file storage).

However, because it is a pre-packaged app, its underlying structure is locked. This means it cannot run completely custom native code (Section 18) or specific third-party libraries that require changes to the core iOS or Android files.

When your app grows and you need those advanced, custom features, you won't get stuck! Instead, you will create a "Development Build". A Development Build is simply your very own, personalized version of the Expo Go app that contains your custom native code. We will cover exactly how to do this later in the course.

Expo Go Development Build Bare React Native
Setup time Minutes ~15–30 min Hours
Custom native code ✗ No ✓ Yes ✓ Yes
Good for learning ✓ Excellent ✓ Good ✗ Steep
Full control ✗ Limited ✓ Yes ✓ Full

Why this course uses Expo

  • You'll be writing real app code within the first ten minutes, not fighting build tools.
  • The entire industry has largely converged on Expo for new React Native projects, including at companies with dedicated mobile teams.
  • Everything you learn transfers directly — Expo apps are React Native apps; nothing is "fake" or "training wheels only."
Beginner mistakeThinking Expo and React Native are competing frameworks. Expo is built on top of, and ships, React Native.
Intermediate mistakeAdding a native library that needs custom native code, then being confused why it "doesn't work" in Expo Go.
Professional mistakeAvoiding Expo entirely for a production app "for control," without weighing the real maintenance cost of managing native build tooling yourself.

Project Structure

Open your new project folder and you'll see a number of files and folders. Here's what actually matters at this stage: 

my-first-app/

│   +-- index.tsx         # the first screen users see
│   +-- _layout.tsx       # shared layout/navigation config
+-- assets/               # images, fonts, icons

+-- tsconfig.json         # TypeScript configuration

The files that matter most right now

File Purpose
app/index.tsx The component rendered for your app's home screen.
app/_layout.tsx Wraps every screen — good place for shared navigation setup, fonts, or providers.
package.json Lists every library your project depends on, and defines scripts like npm start.
app.json Configures your app's name, icon, splash screen, and platform-specific settings.
Why a folder-based structure? Expo Router (Section 5) turns your folder structure inside app/ directly into your navigation structure — a file named app/profile.tsx automatically becomes a screen reachable at /profile. This is why the folder layout matters more in RN+Expo than in a typical single-page setup.
Beginner mistakeEditing files inside auto-generated folders like node_modules — never edit these; they're regenerated and your changes will be lost.
Intermediate mistakePutting every component directly in app/, which conflates "screens" with "reusable pieces" and creates unintended routes.

Fetching Data from an API

Almost every real app needs data from a server — a list of products, messages, or posts. React Native uses the same fetch API you may have seen on the web, combined with React's useState and useEffect hooks.

app/index.tsx 
import { useEffect, useState } from 'react';
import { Text, View, ActivityIndicator } from 'react-native';

export default function App() {
  const [posts, setPosts] = useState([]);
  const [loading, setLoading] = useState(true);

  useEffect(() => {
    fetch('https://jsonplaceholder.typicode.com/posts')
      .then((res) => res.json())
      .then((data) => {
        setPosts(data);
        setLoading(false);
      });
  }, []);

  if (loading) return <ActivityIndicator />;

  return (
    <View>
      <Text>Loaded {posts.length} posts!</Text>
    </View>
  );
}
  • useState([]) — creates a piece of state (posts) that starts empty and will hold the data once it arrives.
  • useEffect(() => {...}, []) — runs the code inside once, right after the component first appears. The empty array [] means "don't re-run this."
  • fetch(url) — makes an HTTP request. It's asynchronous, meaning the rest of the code doesn't wait around for it.
  • .then((res) => res.json()) — the response arrives as raw data; .json() parses it into a usable JavaScript object/array.
  • If you removed the [] dependency array, this fetch would re-run on every re-render — an easy way to accidentally hammer an API.

Intermediate mistakeNot handling network errors — a real app must handle "no internet" or "server error" gracefully, not just the happy path.
Professional mistakeFetching directly inside components everywhere instead of centralizing data-fetching logic (e.g., with React Query), leading to duplicated loading/error handling across the app.

Quick check

Why does the code pass an empty array [] as the second argument to useEffect?

Correct. The dependency array controls when the effect re-runs. Empty means "only on first mount" — perfect for a one-time data fetch.

Rendering Lists and TypeScript

Showing one item is easy. Real apps need to show many — a feed, a chat history, a product catalog. React Native provides FlatList, purpose-built for rendering large lists efficiently.

Why not just use .map() in a View? You can, for short lists — but FlatList only renders items currently visible on screen ("virtualization"), which matters enormously for performance once a list has hundreds or thousands of items.
app/index.tsx 
import { FlatList, Text, View } from 'react-native';

// TypeScript: describe the shape of a Post
type Post = {
  id: number;
  title: string;
};

export default function PostList({ posts }: { posts: Post[] }) {
  return (
    <FlatList
      data={posts}
      keyExtractor={(item) => String(item.id)}
      renderItem={({ item }) => (
        <View style={{ padding: 12 }}>
          <Text>{item.title}</Text>
        </View>
      )}
    />
  );
}
  • type Post = {...} — TypeScript lets you describe exactly what shape your data has. If you accidentally typed item.titel, TypeScript would catch it before you even run the app.
  • data={posts} — the array FlatList will render.
  • keyExtractor — tells React how to uniquely identify each item, so it can efficiently update the list when data changes.
  • renderItem — a function describing how each individual item should look.

Why TypeScript, specifically?

  • Catches mistakes early — typos and wrong data shapes are flagged while you type, not after your app crashes.
  • Self-documenting — a function's types tell you exactly what it expects, without reading its whole implementation.
  • Industry standard — the vast majority of professional React Native codebases use TypeScript by default (Expo projects use it out of the box).
Beginner mistakeForgetting keyExtractor — React will warn you, and list re-renders may behave unpredictably.
Intermediate mistakeUsing .map() inside a ScrollView for a genuinely long list, causing every single item to render at once and hurting performance.
Professional mistakeOver-typing everything with any "to move fast," which quietly defeats the entire purpose of using TypeScript.

Working with Images

React Native's Image component handles both local images (bundled with your app) and remote images (loaded from a URL) — but they're used slightly differently.

app/index.tsx 
import { Image } from 'react-native';

// Local image — path is resolved at build time
<Image source={require('../assets/logo.png')} style={{ width: 100, height: 100 }} />

// Remote image — must always specify width/height
<Image
  source={{ uri: 'https://picsum.photos/200' }}
  style={{ width: 200, height: 200, borderRadius: 12 }}
/>
  • require('../assets/logo.png') — bundles the image into your app at build time; the path must be static (you can't build it dynamically with a variable).
  • source={{ uri: ... }} — for remote images, React Native has no way to know the image's dimensions ahead of time, so you must specify width/height yourself, or the image won't appear.
Performance tip For lists of remote images, consider expo-image instead of the core Image component — it adds disk caching, smoother loading transitions, and better memory handling out of the box.
Beginner mistakeForgetting to set width/height on a remote image and wondering why nothing shows up.
Intermediate mistakeLoading full-resolution images (e.g., 4000px wide) for a 100px thumbnail, wasting memory and bandwidth.

Styling Your App

There's no CSS file in React Native. Instead, you style components using JavaScript objects, either written inline or organized with StyleSheet.create. The property names are very close to CSS, but written in camelCase.

app/index.tsx 
import { StyleSheet, Text, View } from 'react-native';

export default function Card() {
  return (
    <View style={styles.card}>
      <Text style={styles.title}>Hello!</Text>
    </View>
  );
}

const styles = StyleSheet.create({
  card: {
    padding: 16,
    borderRadius: 12,
    backgroundColor: '#ffffff',
    shadowColor: '#000',
    shadowOpacity: 0.1,
  },
  title: {
    fontSize: 18,
    fontWeight: '600',
  },
});
Hello!

Play with the styles

Why StyleSheet.create instead of plain objects?

  • Slightly better performance — styles are given IDs and can be referenced instead of recreated every render.
  • Editor autocomplete and type-checking for valid style properties.
  • Keeps components readable by separating layout logic from style definitions.
Beginner mistakeUsing web CSS property names like background or margin: '10px' — React Native uses backgroundColor and plain numbers (no "px").
Intermediate mistakeMassive inline style objects recreated on every render instead of using StyleSheet.create or extracting constants.

Navigation Basics

A real app has more than one screen. Expo Router (covered in full depth in Section 5) gives you file-based navigation: the files inside your app/ folder automatically become your app's screens and routes. 

app/
+-- index.tsx        ?  "/"       (Home screen)
+-- profile.tsx      ?  "/profile"
+-- settings.tsx     ?  "/settings"
app/index.tsx 
import { Link } from 'expo-router';
import { Text, View } from 'react-native';

export default function Home() {
  return (
    <View>
      <Link href="/profile">Go to Profile</Link>
    </View>
  );
}
  • import { Link } from 'expo-router' — similar to a web <a> tag, but for native screen navigation.
  • href="/profile" — matches the filename app/profile.tsx automatically. No manual route configuration needed.
Why file-based routing? It mirrors how Next.js handles routing on the web, so if you've used that, this feels immediately familiar. It also means your file structure is your app map — you can understand an app's navigation just by looking at its folders.
Beginner mistakeManually trying to configure routes/screens the way older "React Navigation only" tutorials show, when Expo Router handles this via the file system.
Intermediate mistakeNot understanding the difference between pushing a new screen vs. replaceing the current one — covered in full in Section 5.
