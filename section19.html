Apple Widgets

iOS home screen widgets are built with WidgetKit — a separate mini-app "extension" that runs independently of your main app, refreshing on a timeline rather than staying constantly live.

Click each size to preview it

??
72—F

Small: one glanceable stat — a single piece of information, no interaction.

targets/widget/Widget.swift 
import WidgetKit
import SwiftUI

struct WeatherWidget: Widget {
  var body: some WidgetConfiguration {
    StaticConfiguration(kind: "WeatherWidget", provider: Provider()) { entry in
      Text("\(entry.temperature)—")
    }
    .configurationDisplayName("Weather")
    .supportedFamilies([.systemSmall, .systemMedium])
  }
}
  • StaticConfiguration — the simplest widget type: no user-configurable options, just data that changes over time.
  • provider: Provider() — a small type you define that tells WidgetKit when to fetch new data and what the "timeline" of future updates looks like.
  • .supportedFamilies — declares which sizes this widget supports; the same view logic typically renders differently per size, as you saw in the preview above.
The key constraint to internalize A widget is not your running React Native app in miniature — it's a separate, lightweight native extension that can't run your JS bundle. Data must be shared between your app and the widget via a mechanism like an App Group's shared storage, updated periodically, not live-streamed.
Beginner mistakeExpecting a widget to update instantly when the main app's state changes — it only refreshes on its own timeline schedule.
Intermediate mistakeForgetting to configure an App Group, leaving the widget with no way to read data written by the main app.

How to Upgrade Expo SDK

Expo ships a new SDK version roughly every few months, tracking new React Native releases. Skipping upgrades for too long compounds risk — each version bump is manageable; jumping five versions at once often isn't.

Run through the upgrade process yourself
Read changelog
Bump expo package
expo install --fix
Rebuild dev client
Test thoroughly

terminal 
# 1. Update the core expo package to the new SDK version
npx expo install expo@latest

# 2. Let Expo align every other dependency to compatible versions
npx expo install --fix

# 3. Rebuild your development build — native code has changed
eas build --profile development --platform all
  • expo install --fix — scans your package.json and corrects any dependency versions that don't match what the new SDK expects, which is safer than manually guessing compatible versions.
  • A new development build is required after an SDK bump because native code genuinely changed — reusing an old development build against new JS can cause confusing native/JS version mismatches.
Why "one version at a time" beats "skip ahead" Each SDK release's changelog documents breaking changes relative to the immediately previous version. Jumping from SDK 49 to 52 directly means reading and reconciling three changelogs' worth of breaking changes simultaneously — upgrading sequentially (49?50?51?52), testing at each step, isolates problems far more easily.
Beginner mistakeManually editing dependency versions in package.json by guesswork instead of running expo install --fix.
Intermediate mistakeUpgrading the JS/Expo package but forgetting to generate a new development/production build, running new JS against stale native code.
Professional mistakeLetting a production app fall multiple SDK versions behind, turning a routine upgrade into a multi-day migration project.

Quick check

Why upgrade one SDK version at a time rather than jumping straight to the newest?

Correct — skipping versions compounds the number of breaking changes you need to reconcile at once.

Meta Ads & Facebook SDK

If you run Facebook/Instagram ad campaigns to acquire users, the Meta SDK lets you report back which in-app actions (signups, purchases) resulted from those ads — closing the loop between ad spend and real outcomes.

app/checkout.tsx 
import { Settings, AppEventsLogger } from 'react-native-fbsdk-next';

// after a successful purchase:
AppEventsLogger.logPurchase(9.99, 'USD');
  • logPurchase(amount, currency) — a standard, predefined event Meta's attribution system specifically understands, distinct from arbitrary custom events.
  • Using standard events (purchase, signup, add-to-cart) rather than only custom ones gives you access to Meta's built-in ad optimization tools that target users likely to complete that specific action.
The App Tracking Transparency requirement (iOS) On iOS, cross-app attribution like this requires the user's explicit permission via Apple's App Tracking Transparency (ATT) prompt. Without permission granted, event data is still logged but in a limited, aggregated form — you cannot bypass this by not asking, and not asking when required is itself a policy violation.
app/_layout.tsx 
import * as TrackingTransparency from 'expo-tracking-transparency';

const { status } = await TrackingTransparency.requestTrackingPermissionsAsync();
// only enable full Meta attribution if status === 'granted'
Beginner mistakeIntegrating the Facebook SDK without implementing the ATT prompt on iOS — this is both a policy violation and legally required in relevant jurisdictions.
Intermediate mistakeLogging only custom events instead of Meta's standard events, missing out on ad platform features built specifically around recognized event types.
Professional mistakeNot accounting for the significant data limitations when users decline tracking — campaign attribution should degrade gracefully, not assume full data is always available.

Section 19 final check

What must happen before full cross-app attribution data is available on iOS?

Correct — ATT permission is a hard requirement for full attribution data on iOS, not an optional enhancement.
