TOPIC 1 / 4

Development Builds

Recall from Section 2: a development build is your project's own custom version of Expo Go. EAS Build is the cloud service that compiles it for you — no local Xcode/Android Studio setup required, even from a Windows or Linux machine.

eas.json 
{
  "build": {
    "development": {
      "developmentClient": true,
      "distribution": "internal"
    }
  }
}
terminal 
eas build --profile development --platform ios
  • "developmentClient": true — bakes in the dev-client tooling (Section 2), so this build includes the debug menu and can load your JS bundle from your local dev server.
  • "distribution": "internal" — the resulting file installs directly on a registered test device, without going through the App Store or Play Store.
  • --profile development — tells EAS Build which named configuration from eas.json to use for this run.

Where this fits in the bigger picture

Click each profile below to see its typical configuration and purpose — this is the same three-profile structure a real eas.json almost always has.

Development: Includes dev-client tooling, installs directly on your own registered device, used for day-to-day feature work with live JS reload.
Beginner mistakeSending a development build to a non-technical tester expecting a "final" app experience — it includes debug tooling not meant for end users.
TOPIC 2 / 4

Preview Builds

A preview build is a release-like build (no debug tooling, no dev server dependency) that isn't yet submitted to an app store — perfect for handing to a designer, stakeholder, or QA team member to try on their own device.

eas.json 
{
  "build": {
    "preview": {
      "distribution": "internal",
      "channel": "preview"
    }
  }
}
  • Unlike development builds, preview builds don't include dev-client debug tooling — they behave like the real, final app.
  • "channel": "preview" — connects this build profile to a specific EAS Update channel (Section 14), so OTA updates can target testers separately from production users.
Real-world workflow A common pattern: every pull request automatically triggers a preview build via EAS Workflows (Section 15), giving reviewers an actual installable app to test the exact change — not just reading a code diff.
Intermediate mistakeForgetting that iOS internal distribution requires registering each tester's device UDID in advance — unlike Android, which can sideload freely.

eas.json 
{
  "build": {
    "production": {

terminal 
eas build --profile production --platform all
  • --platform all — queues both an iOS and Android build in one command, run in parallel on Expo's build infrastructure.

Quick check

What's the key functional difference between a preview build and a production build?

Correct — both are release-like (no debug tooling), but only production builds go through app store submission.
TOPIC 4 / 4

Automating App Version Code

Every app store submission needs a unique, ever-increasing internal build number (iOS calls it buildNumber, Android calls it versionCode) — separate from your human-readable version like "2.3.0". Forgetting to bump it manually is one of the most common, avoidable submission failures.

Try it: simulate autoIncrement Each "Trigger production build" press below simulates what "autoIncrement": true does automatically on every real build — bumping the internal build number without you touching any config file.
app.json
version: "2.3.0" (unchanged)
buildNumber: 12

What's happening

Your human-facing version stays the same across many builds — you only bump it for a real user-facing release. The internal build number increments on every single build, since app stores require every submitted binary to have a unique number.

eas.json 
{
  "build": {
    "production": {
      "autoIncrement": true
    }
  },
  "cli": {
    "appVersionSource": "remote"
  }
}
  • "autoIncrement": true — on every build using this profile, EAS bumps the build number for you before compiling.
  • "appVersionSource": "remote" — tells EAS to track the current version/build number on Expo's servers rather than solely inside your local app.json, so it stays correct even across different machines/CI runners on a team.
Beginner mistakeManually editing buildNumber in app.json before every submission and occasionally forgetting, causing an instant app store rejection for a duplicate build number.
Intermediate mistakeUsing autoIncrement without "appVersionSource": "remote" on a team, where different machines' local counters drift out of sync.

Section 12 final check

Why must the internal build number increase on every single submission, even for a tiny bug fix?

Correct — this is an app store requirement, independent of your semantic version string, which is why automating it removes an entire class of avoidable submission failures.
