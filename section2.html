TOPIC 2 / 6

Why Expo

Section 1 introduced Expo Go as a way to preview apps instantly. Here we zoom out: why does the React Native ecosystem, as a whole, now default to Expo for nearly every new project — including at large companies?

The problem before Expo existed

Historically, "bare" React Native projects meant every team independently solved the same problems: how do we handle app icons across densities, how do we manage build numbers, how do we configure push notification certificates, how do we keep native dependencies from breaking on every OS update. Expo standardizes all of this into shared, maintained tooling.

??
Consistent APIs

One expo-camera, one expo-location — instead of picking between several competing community libraries with different quirks.

???
Managed native code

Native project files (the ios/ and android/ folders) can be auto-generated from config, instead of hand-maintained and merge-conflicted forever.

??
Cloud infrastructure

EAS Build, Submit, and Update (Sections 12–15) remove the need to personally maintain Mac build machines or CI pipelines from scratch.

An important nuance Choosing Expo does not lock you out of custom native code. Through "config plugins" and development builds, you can add virtually any native capability a bare React Native project could — you're opting into better defaults, not a restricted subset.
Concern Bare React Native Expo
New project setup Manual, platform-specific One CLI command
OTA updates Build your own or third-party Built-in via EAS Update (Section 14)
Native module APIs Pick from many community libraries Curated, consistent Expo SDK
Custom native code Always available Available via dev builds / config plugins
Beginner mistakeBelieving "Expo apps aren't real apps" or can't be published to app stores — they can, and are, by large companies.
Professional mistakeMigrating a mature bare RN app to Expo without first auditing which native modules have Expo-compatible equivalents or config plugins.

Quick check

Does choosing Expo prevent you from ever writing custom native (Swift/Kotlin) code?

Correct. Expo Go specifically can't run custom native code, but a development build (your own custom Expo Go) absolutely can — this is covered in Section 18: Native Modules.
TOPIC 3 / 6

Android Environment Setup

This gives you an Android emulator on your own computer — useful for testing without a physical device, and required once you need custom native code.

Step by step

  1. Download and install Android Studio from developer.android.com.
  2. During setup, make sure the Android SDK, Android SDK Platform, and Android Virtual Device components are all checked.
  3. Open Android Studio ? More Actions ? Virtual Device Manager ? create a new device (a recent Pixel profile with a recent Android system image is a safe default).
  4. Set the ANDROID_HOME environment variable so command-line tools can find your SDK.
~/.zshrc or ~/.bashrc 
# macOS/Linux example — adjust path if your SDK lives elsewhere
export ANDROID_HOME=$HOME/Library/Android/sdk
export PATH=$PATH:$ANDROID_HOME/emulator
export PATH=$PATH:$ANDROID_HOME/platform-tools
  • ANDROID_HOME — tells build tools and the CLI where your Android SDK is installed, so commands like adb and emulator work from any terminal.
  • After editing this file, restart your terminal (or run source ~/.zshrc) for the changes to take effect.
terminal 
# with an emulator running, from your project folder:
npx expo start --android
Don't have a powerful computer? Emulators are RAM and CPU-hungry. If your machine struggles, skip this and keep testing on a real Android device via Expo Go — everything in this course works either way, until Section 18 (Native Modules).
Beginner mistakeForgetting to set ANDROID_HOME, causing "command not found" or "SDK not found" errors.
Intermediate mistakeCreating a virtual device with a very old Android version that doesn't match your app's minimum target, causing confusing crashes.
  • Beginner Create an Android Virtual Device and successfully run your Section 1 project on it.
TOPIC 4 / 6

iOS Environment Setup

Mac required Building for the iOS Simulator or installing on physical iPhones with custom native code requires a Mac — this is an Apple platform restriction, not a React Native/Expo limitation. Windows/Linux users can still develop iOS apps using Expo Go on a real iPhone, or EAS Build's cloud iOS builds (Section 12).

Step by step (on a Mac)

  1. Install Xcode from the Mac App Store (it's large — several GB — so start this early).
  2. Open Xcode once fully, and accept its license agreement.
  3. Install the Xcode Command Line Tools if prompted.
  4. Open Xcode ? Settings ? Platforms, and confirm an iOS Simulator runtime is installed.
terminal 
# confirm Xcode command line tools are set up correctly
xcode-select --install

# from your project folder, with Xcode installed:
npx expo start --ios
  • xcode-select --install — installs the command-line developer tools Expo's CLI needs to talk to the Simulator, separate from the full Xcode app.
  • npx expo start --ios — builds and opens your app directly in the iOS Simulator, no QR code needed.
Scenario What you need
Testing on your own iPhone Expo Go app (any OS on your computer)
Testing on iOS Simulator A Mac with Xcode
Custom native code on iOS A Mac with Xcode, or EAS Build's cloud service
Beginner mistakeStarting Xcode's multi-GB download the moment you need it, rather than in advance — plan for it to take a while on slower connections.
Intermediate mistakeAssuming you're fully blocked from iOS development on Windows/Linux — EAS Build's cloud builds remove that requirement for anything beyond local Simulator testing.
TOPIC 5 / 6

RN New Architecture

Section 1 briefly compared the old Bridge architecture to the New Architecture. Here we go one level deeper, because as of recent Expo SDK versions, every new project uses the New Architecture by default — understanding it is no longer optional trivia.

The three pieces of the New Architecture

JSI
JavaScript Interface

Lets JavaScript hold direct references to native objects and call native methods synchronously when needed — replacing the old async-only bridge.

Fab
Fabric

The new rendering system. Builds the UI tree using C++ shared between platforms, reducing inconsistency between iOS and Android rendering.

TM
TurboModules

Native modules are now loaded lazily (only when first used) instead of all upfront, speeding up app startup time.

Why should a beginner care?

You won't write JSI or Fabric code directly. But you will:

  • See "New Architecture" mentioned in library documentation and know what it means.
  • Understand why some older libraries needed updates to remain compatible.
  • Understand why startup performance and heavy UI interactions generally feel faster than they did historically.
Practical takeaway When a library's docs say "requires New Architecture" or "not yet compatible with New Architecture," this is now the concept they're referring to. Expo-managed projects handle this compatibility layer for you in most cases.
Intermediate mistakeDisabling the New Architecture "to fix a bug" without first checking whether the actual issue was an outdated library version.
Professional mistakeDeep-diving into writing custom TurboModules before understanding whether a simpler Expo Modules API (Section 18) solves the same problem.

Quick check

What does JSI primarily remove the need for, compared to the old Bridge?

Correct — the old bridge required converting every call into JSON and passing it asynchronously. JSI allows direct references and calls, cutting that overhead.
TOPIC 6 / 6

Expo Go vs Development Builds

This is the single most common point of confusion for people new to Expo, so let's make it completely unambiguous.

??
Expo Go

A single app, published once by Expo, pre-installed with the entire standard Expo SDK's native code. You just install it from the App/Play Store.

???
Development Build

Your own custom version of Expo Go, built specifically for your project, containing exactly the native code your project needs — including custom modules.

Expo Go Development Build
Install source App Store / Play Store Built by you via EAS Build or locally
Custom native modules ✗ Not supported ✓ Fully supported
Setup effort None — already installed One build, then reused like Expo Go
Good for Learning, prototyping, standard SDK features Anything needing custom native code
terminal 
# install the dev-client package once your project needs it
npx expo install expo-dev-client

# build a development build for a connected device/simulator
npx expo run:ios
# or
npx expo run:android
  • expo-dev-client — the package that turns your project into "its own Expo Go," including a small in-app menu for reloading and debugging.
  • npx expo run:ios / run:android — compiles your project's native code locally and installs the resulting development build on a simulator/device.
How to decide, in practice Stay on Expo Go for as long as every library you use is compatible with it (most are, for standard app features). The moment you add a library that says "requires custom native code" or "not compatible with Expo Go," that's your signal to switch to a development build — a one-time setup, not a full rewrite.
Beginner mistakeAssuming a development build means "leaving Expo" — it doesn't; it's still a fully Expo-managed project, just with your own native shell.
Intermediate mistakeRebuilding a development build for every small JS change — you only need to rebuild it when native dependencies change, not for everyday JS edits.

Section 2 final check

When should you switch from Expo Go to a development build?

Correct. Expo Go covers the standard SDK well; a development build becomes necessary specifically when you need native code beyond what Expo Go ships with.
