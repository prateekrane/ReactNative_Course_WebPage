Introduction to Native Modules

Every time you've used a feature like the camera, notifications, or secure storage, you've been using a native module — JS-callable code backed by real Swift/Kotlin. The Expo Modules API is the modern, batteries-included way to write your own.

Click each layer to see how a call actually travels

1. JS call: Your code calls something like await Flashlight.toggle() — an ordinary-looking async function call, no different from calling any other JS function.
Why write one at all? Reach for a native module only when: (1) no existing Expo SDK or community library already does it, and (2) the capability genuinely requires platform APIs unavailable from pure JS — things like custom Bluetooth protocols, specialized camera processing, or OS-level widgets (Section 19).
Layer What it does
JS call Your ordinary-looking async function call in React Native code
JSI Passes the call and arguments directly to native code, no JSON serialization needed
Expo Module Your Swift/Kotlin code, registered and exposed via the Expo Modules API
OS API The actual platform capability being used — camera, sensors, file system, etc.
Beginner mistakeWriting a custom native module for something an existing, well-maintained Expo SDK package or community library already solves.
Intermediate mistakeAssuming native modules require the old Bridge — modern Expo Modules are built on JSI/TurboModules from the New Architecture (Section 2) by default.

Create a Native Module

Let's build a real, minimal native module: a flashlight toggle. Watch each creation stage animate below, then look at the actual code for each.

Write Swift function
Write Kotlin function
Generate JS binding
Call from app

terminal 
npx create-expo-module flashlight --local
  • --local — scaffolds the module directly inside your existing project's modules/ folder, ideal for app-specific native code rather than a publishable standalone package.
modules/flashlight/ios/FlashlightModule.swift 
import ExpoModulesCore
import AVFoundation

public class FlashlightModule: Module {
  public func definition() -> ModuleDefinition {
    Name("Flashlight")

    AsyncFunction("toggle") { () -> Bool in
      guard let device = AVCaptureDevice.default(for: .video) else { return false }
      try device.lockForConfiguration()
      device.torchMode = device.torchMode == .on ? .off : .on
      device.unlockForConfiguration()
      return device.torchMode == .on
    }
  }
}
  • Name("Flashlight") — registers how this module will be addressed from JS, e.g. requireNativeModule('Flashlight').
  • AsyncFunction("toggle") — declares a function callable from JS as a Promise-returning async call — matching JS's own async/await model.
  • The actual torch-toggling logic uses Apple's AVFoundation — a real system API with no React Native equivalent exposed by default.
modules/flashlight/android/FlashlightModule.kt 
import expo.modules.kotlin.modules.Module
import expo.modules.kotlin.modules.ModuleDefinition

class FlashlightModule : Module() {
  override fun definition() = ModuleDefinition {
    Name("Flashlight")

    AsyncFunction("toggle") {
      val cameraManager = appContext.reactContext?.getSystemService(Context.CAMERA_SERVICE) as CameraManager
      // toggle torch via cameraManager.setTorchMode(...)
      true
    }
  }
}
  • Notice the structural mirror to the Swift file — Name(...) and AsyncFunction(...) are the same Expo Modules API concepts, just Kotlin syntax on Android's side.
app/index.tsx 
import Flashlight from 'flashlight';

const isOn = await Flashlight.toggle();
The payoff From the app's perspective, this is indistinguishable from calling any other async function — all the Swift/Kotlin complexity is completely hidden behind one clean JS API.
Beginner mistakeForgetting to implement both the iOS and Android sides — a module missing one platform's implementation will crash or no-op there.
Intermediate mistakeDoing expensive/blocking work inside a native function without it being properly async, freezing the native UI thread.

Quick check

What does Name("Flashlight") do inside a module's definition?

Correct — it's purely an internal identifier connecting the JS side to this specific native module.

Create a Native View

Beyond function calls, native modules can also expose entire views — real native UI, rendered inline in your React Native tree, exactly like the custom SwiftUI/Compose views from Section 7, but built through the plain native module system.

modules/flashlight/ios/FlashlightView.swift 
import ExpoModulesCore

class TorchIndicatorView: ExpoView {
  let onTorchChange = EventDispatcher()

  required init(appContext: AppContext? = nil) {
    super.init(appContext: appContext)
    backgroundColor = .black
  }
}
  • ExpoView — the base class for any custom native view exposed through Expo Modules; it slots directly into React Native's view hierarchy.
  • EventDispatcher() — lets this native view fire events back up to JS (like an onTorchChange prop callback), the reverse direction of the function calls from the previous topic.
modules/flashlight/ios/FlashlightModule.swift (registering the view) 
View(TorchIndicatorView.self) {
  Events("onTorchChange")
}
app/index.tsx 
import { requireNativeViewManager } from 'expo-modules-core';

const TorchIndicatorView = requireNativeViewManager('Flashlight');

<TorchIndicatorView onTorchChange={(e) => console.log(e.nativeEvent)} style={{ width: 40, height: 40 }} />
How this differs from Section 7's Expo UI Expo UI (Section 7) gives you pre-built components that already wrap SwiftUI/Compose for common needs. This lower-level native view system is what you reach for when you need something fully custom that neither Expo UI nor any existing library provides.
Intermediate mistakeForgetting to register the view with View(...) inside the module definition — without it, requireNativeViewManager won't find anything on the JS side.

Native Modules Next Steps

Before reaching for a custom native module on your next project, run through this decision checklist — it'll save you significant time in most cases.

Where to go from here

  • Search first. Check the Expo SDK and popular community libraries (via a quick search of the React Native directory) before writing anything custom.
  • Start local. Use --local modules (this section's approach) before extracting a standalone published package — you can always extract later once the API stabilizes.
  • Read real examples. Expo's own SDK packages are open source — reading how expo-camera or expo-notifications are built is one of the best ways to learn real module patterns.
  • Revisit Section 7. If your need is UI-shaped (a native-feeling control, not a system capability), Expo UI's higher-level components may already cover it without writing raw native view code.
A healthy default Most successful React Native apps ship for years without ever writing a custom native module — reach for one only when you've confirmed, concretely, that nothing else solves the problem.
Professional mistakeTreating native module authorship as a rite of passage rather than a genuine, justified engineering need — it adds real maintenance burden (two more languages, two more build targets) that should be earned, not defaulted to.

What should you check before writing a custom native module?

Correct — custom native modules add real long-term maintenance cost, so it's worth confirming nothing existing already covers the need.
