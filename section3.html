Components and APIs

Every screen you'll ever build is made of components. Here we go from the small set of built-in building blocks, to writing your own reusable components, to advanced patterns used in production codebases.

Basic Components

React Native ships a small set of built-in components that map directly to native UI elements. You already met two of them (View and Text) in Section 1 — here's the full core set you'll reach for constantly.

?
View

An invisible container, like a <div> on the web. Used to group and lay out other components.

Tt
Text

The only component that can display text. Text must always be wrapped in a Text, never placed directly in a View.

?
ScrollView

A container that renders all its children immediately, but lets the user scroll if content overflows the screen.

?
FlatList

Efficiently renders long lists by only drawing what's on screen (met in Section 1, Topic 8).

?
TextInput

Lets the user type text — the native equivalent of an <input>.

?
Pressable

Makes anything tappable, with fine control over press states — the modern replacement for the older TouchableOpacity.

??
Image

Displays local or remote images (met in Section 1, Topic 9).

?
SafeAreaView

Keeps content clear of notches, camera cutouts, and system bars.

A component you haven't seen yet: Pressable

components/LikeButton.tsx 
import { useState } from 'react';
import { Pressable, Text } from 'react-native';

export default function LikeButton() {
  const [liked, setLiked] = useState(false);

  return (
    <Pressable
      onPress={() => setLiked((prev) => !prev)}
      style={({ pressed }) => ({
        opacity: pressed ? 0.6 : 1,
        padding: 10,
      })}
    >
      <Text>{liked ? '?? Liked' : '?? Like'}</Text>
    </Pressable>
  );
}
  • onPress — the function that runs when the user taps. Note it's onPress, not onClick as on the web — there's no mouse, only touch.
  • style={({ pressed }) => ({...})} — Pressable can accept a function as its style, giving you the live pressed state to react to (e.g. dimming on press).
  • If you swapped this for a plain View with no Pressable, tapping it would do nothing — View has no touch handling built in.

Try it

Click the button in the phone — this is exactly the interaction the code above produces.

Pressable vs. TouchableOpacity vs. Button Button is quick but barely stylable. TouchableOpacity is older and still common in existing codebases. Pressable is the current recommended API — more flexible press-state handling, and the direction the ecosystem has standardized on.
Beginner mistakePlacing raw text directly inside a View instead of wrapping it in Text — this throws an error in React Native, unlike the web.
Intermediate mistakeNesting a FlatList inside a ScrollView — both try to manage scrolling, causing broken virtualization and console warnings.
Professional mistakeStill reaching for TouchableOpacity out of habit in new code without considering Pressable's more flexible API.

Quick check

Why must text always be wrapped in a <Text> component?

Correct. Unlike a web <div>, a React Native View cannot render text directly — it has no concept of text rendering, only layout.
  • Beginner Build a Pressable that toggles between "Show" and "Hide" text on tap.
  • Intermediate Add a TextInput above the button that only enables the button once at least 3 characters are typed.
TOPIC 2 / 3

Custom Components

Basic components are your raw materials. A custom component is one you write, by combining basic components into something reusable — a button styled your way, a card layout, a list item design used in ten different screens.

Why bother, instead of repeating the same JSX everywhere?

  • One place to fix bugs. Change the button's padding once, and every screen using it updates.
  • Readable screens. <ProductCard product={item} /> reads far better than 15 lines of raw layout repeated inline.
  • Props make components flexible without duplicating code for every slight variation.
components/PrimaryButton.tsx 
import { Pressable, Text, StyleSheet } from 'react-native';

type Props = {
  label: string;
  onPress: () => void;
  disabled?: boolean;
};

export default function PrimaryButton({ label, onPress, disabled }: Props) {
  return (
    <Pressable
      onPress={onPress}
      disabled={disabled}
      style={({ pressed }) => [
        styles.button,
        disabled && styles.disabled,
        pressed && { opacity: 0.85 },
      ]}
    >
      <Text style={styles.label}>{label}</Text>
    </Pressable>
  );
}

const styles = StyleSheet.create({
  button: { backgroundColor: '#ff5a3c', padding: 14, borderRadius: 10, alignItems: 'center' },
  disabled: { backgroundColor: '#ccc' },
  label: { color: '#fff', fontWeight: '600' },
});
  • type Props — defines exactly what this component accepts. TypeScript will now warn you at every usage site if you forget onPress or misspell label.
  • onPress: () => void — this says "a function that takes no arguments and returns nothing," which is the shape of a typical press handler.
  • disabled?: boolean — the ? makes this prop optional; callers don't have to pass it.
  • style={[styles.button, disabled && styles.disabled, ...]} — React Native accepts an array of styles and merges them left to right, letting you conditionally layer styles.

Using it elsewhere

app/index.tsx 
<PrimaryButton label="Continue" onPress={() => router.push('/next')} />
Real-world example In an e-commerce app, a single ProductCard component might appear on the home screen, in search results, and in a "recently viewed" list — three very different screens, one component, one place to fix a bug or change the design.
Beginner mistakeCopy-pasting the same button JSX across five screens instead of extracting one component — any design change now needs five separate edits.
Intermediate mistakeMaking a component accept a giant, vague style override prop instead of well-named specific props, making it easy to accidentally break its design.
Professional mistakeExtracting a component too early, before a second real usage exists — premature abstraction that guesses wrong about what should be configurable.

Quick check

What does adding ? after a prop name in a TypeScript type do?

Correct. Without ?, TypeScript would require every usage of the component to pass that prop.
  • Beginner Extract a Card component that wraps any children in padded, rounded, white background styling.
  • Intermediate Give PrimaryButton a variant prop ("primary" | "secondary") that changes its colors.
TOPIC 3 / 3

Advanced Custom Components

Once you're comfortable extracting simple components, the next level is composition patterns that keep components flexible without becoming a tangle of boolean props.

Pattern 1: Composition with children

Instead of a component that accepts many specific props for every possible piece of content, let it accept arbitrary children — the same pattern View itself uses.

components/Card.tsx 
import { View, StyleSheet } from 'react-native';
import { ReactNode } from 'react';

export default function Card({ children }: { children: ReactNode }) {
  return <View style={styles.card}>{children}</View>;
}

const styles = StyleSheet.create({
  card: { backgroundColor: '#fff', borderRadius: 12, padding: 16 },
});

// Usage — Card doesn't need to know what's inside it:
// <Card>
//   <Text>Anything at all</Text>
//   <PrimaryButton label="Go" onPress={...} />
// </Card>

Pattern 2: Compound components

For components with several related parts (like a custom list item with an icon, title, and action), exposing them as named sub-components keeps usage readable and flexible.

components/ListItem.tsx 
import { View, Text } from 'react-native';

function ListItem({ children }: { children: React.ReactNode }) {
  return <View style={{ flexDirection: 'row', alignItems: 'center', padding: 12 }}>{children}</View>;
}
ListItem.Icon = ({ children }: { children: React.ReactNode }) => <View style={{ marginRight: 12 }}>{children}</View>;
ListItem.Title = ({ children }: { children: React.ReactNode }) => <Text style={{ fontWeight: '600' }}>{children}</Text>;

export default ListItem;

// Usage:
// <ListItem>
//   <ListItem.Icon><Avatar /></ListItem.Icon>
//   <ListItem.Title>Jane Doe</ListItem.Title>
// </ListItem>
  • ListItem.Icon = (...) — JavaScript functions are objects, so you can attach properties to them. This groups related sub-components under one clear namespace.
  • This pattern shines when a component has a fixed "slot" structure (icon, title, action) rather than fully arbitrary children.

Pattern 3: Forwarding refs for reusable inputs

When wrapping a native component like TextInput, sometimes a parent screen needs to directly call methods on it (like focusing it). forwardRef makes that possible through your custom wrapper.

components/FormInput.tsx 
import { forwardRef } from 'react';
import { TextInput, TextInputProps } from 'react-native';

const FormInput = forwardRef<TextInput, TextInputProps>((props, ref) => {
  return <TextInput ref={ref} style={{ borderWidth: 1, borderRadius: 8, padding: 10 }} {...props} />;
});

export default FormInput;
When to reach for each pattern Use children composition for generic wrappers (cards, screens). Use compound components when there's a clear fixed structure with named parts. Use forwardRef specifically when a parent genuinely needs imperative access (focus, scroll-to, play/pause) — not as a default habit.
Beginner mistakeTrying to pass a ref to a custom component that doesn't use forwardRef — React will warn that function components can't be given refs directly.
Intermediate mistakeOverusing compound components for simple cases where a couple of plain props would be clearer and easier to type-check.
Professional mistakeBuilding elaborate compound-component APIs for internal, single-use components — the pattern earns its complexity in shared design-system components, not one-off screens.

Section 3 final check

Why would you use forwardRef in a custom input component?

Correct — forwardRef exists specifically for cases where a parent needs direct, imperative access to the wrapped native element.
  • Intermediate Build a compound Accordion / Accordion.Item pair of components.
  • Advanced Build a FormInput with forwardRef, then use a parent's ref to auto-focus it when a screen loads.
