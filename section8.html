Testing

Two complementary layers of testing: fast unit tests for your logic and components with Jest, and full end-to-end tests that drive a real app with Maestro.

Unit Test with Jest

A unit test checks one small piece of logic in isolation — a function, a hook, a component's rendered output — without needing a real device or simulator. Jest is the standard test runner for React Native projects, pre-configured in Expo projects.

utils/formatPrice.ts 
export function formatPrice(cents: number): string {
  return '$' + (cents / 100).toFixed(2);
}
utils/formatPrice.test.ts 
import { formatPrice } from './formatPrice';

describe('formatPrice', () => {
  it('formats whole dollar amounts correctly', () => {
    expect(formatPrice(500)).toBe('$5.00');
  });

  it('formats amounts with cents correctly', () => {
    expect(formatPrice(1999)).toBe('$19.99');
  });
});
  • describe — groups related tests together under a readable label; purely organizational.
  • it (or its alias test) — defines one individual test case with a plain-English description of what it verifies.
  • expect(...).toBe(...) — an assertion: the test fails loudly if the actual value doesn't match the expected one.
  • If you removed describe, the tests would still run — it's purely for grouping/readability in test output, not functionality.
terminal 
npx jest
# or, to re-run automatically on file changes:
npx jest --watch

Testing a component with React Native Testing Library

components/LikeButton.test.tsx 
import { render, fireEvent, screen } from '@testing-library/react-native';
import LikeButton from './LikeButton';

it('toggles text when pressed', () => {
  render(<LikeButton />);
  expect(screen.getByText('?? Like')).toBeTruthy();

  fireEvent.press(screen.getByText('?? Like'));
  expect(screen.getByText('?? Liked')).toBeTruthy();
});
  • render(<LikeButton />) — mounts the component in a test environment, without needing a real device.
  • getByText(...) — queries the rendered output the way a real user would identify it (by visible text), rather than by internal implementation details.
  • fireEvent.press(...) — simulates a real tap on that element.
Guiding principle: test behavior, not implementation Querying by visible text/labels (like a user would) rather than internal component structure means your tests keep passing through internal refactors, and only break when the actual user-facing behavior changes — which is exactly when you want to know.
Beginner mistakeTesting implementation details (like internal state variable names) instead of what the user actually sees and does.
Intermediate mistakeWriting zero tests for logic-heavy utility functions (pricing, date formatting) — exactly the kind of pure functions that are cheapest and most valuable to unit test.
Professional mistakeChasing 100% code coverage as a goal in itself, rather than focusing tests on business-critical logic and common failure points.

Quick check

Why query a rendered component by its visible text rather than internal props/state?

Correct — this is the core philosophy behind React Native Testing Library: test behavior a user would notice, not implementation details.
  • Beginner Write a unit test for a capitalize(str) utility function you write yourself.
  • Intermediate Write a component test for the PrimaryButton from Section 3 that verifies onPress fires exactly once per tap.

E2E with Maestro

Unit tests check pieces in isolation. End-to-end (E2E) tests drive your actual, fully-built app — tapping real buttons, typing into real inputs, verifying real screens appear — the same way a human tester would. Maestro is a popular, simple-to-write E2E tool for mobile apps.

.maestro/login-flow.yaml 
appId: com.yourcompany.yourapp
---
- launchApp
- tapOn: "Email"
- inputText: "test@example.com"
- tapOn: "Password"
- inputText: "password123"
- tapOn: "Log In"
- assertVisible: "Welcome back"
  • Maestro flows are written in plain YAML — no code required, which makes them approachable even for QA team members who don't write JavaScript.
  • tapOn: "Email" — finds and taps an element by its visible text or accessibility label, mirroring how a real user identifies it.
  • assertVisible: "Welcome back" — the test fails if this text isn't found on screen after the login attempt, catching real, full-stack regressions (UI + navigation + API + auth all actually working together).
terminal 
maestro test .maestro/login-flow.yaml
Unit tests (Jest)E2E tests (Maestro)
SpeedVery fast (milliseconds each)Slower (seconds, runs a real app)
What it catchesLogic bugs in isolationReal integration issues across the whole stack
How many you should haveMany, for logic-heavy codeFewer, for critical user flows (login, checkout, etc.)
Real-world example A typical production app might have hundreds of fast Jest unit tests, but only a handful of Maestro E2E flows — covering exactly the flows that would be catastrophic if broken: sign-up, login, checkout, and maybe one or two core features.
Beginner mistakeTrying to E2E-test every possible interaction instead of focusing on critical user journeys — E2E suites become slow and brittle if over-used.
Intermediate mistakeWriting Maestro flows that depend on real, mutable backend data (like a specific existing user), making tests flaky when that data changes.
Professional mistakeNot running E2E tests in CI before releases, catching regressions only after users report them.

Section 8 final check

Which type of test would catch a bug where login succeeds but the home screen fails to load afterward?

Correct — this is exactly the kind of full-stack, cross-screen issue unit tests can't catch in isolation, but E2E tests are designed to.
  • Intermediate Write a Maestro flow that adds an item to a cart and asserts the cart badge count updates.
