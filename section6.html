Animations & Gestures

Animations and Gestures

The modern standard for both is Reanimated (for animations) paired with React Native Gesture Handler (for touch/gestures) — both run animation logic on a separate thread from your JS, keeping motion smooth even if your JS thread is briefly busy.

components/FadeInBox.tsx 
import { useEffect } from 'react';
import Animated, { useSharedValue, useAnimatedStyle, withTiming } from 'react-native-reanimated';

export default function FadeInBox() {
  const opacity = useSharedValue(0);

  useEffect(() => {
    opacity.value = withTiming(1, { duration: 500 });
  }, []);

  const animatedStyle = useAnimatedStyle(() => ({
    opacity: opacity.value,
  }));

  return <Animated.View style={[{ width: 100, height: 100, backgroundColor: 'var(--coral)' }, animatedStyle]} />;
}
  • useSharedValue(0) — creates a value that lives on the UI thread, not React state — reading/writing it doesn't trigger a React re-render.
  • withTiming(1, {...}) — animates the shared value from its current value to 1 over 500ms, running entirely on the native side.
  • useAnimatedStyle — derives a style object from shared values, recalculated on the UI thread every frame.

Try it

Why not just animate with React state? Animating via setState on every frame re-renders your whole component 60 times a second — expensive, and vulnerable to dropped frames if the JS thread is busy. Reanimated's shared values skip React entirely for the animation itself.
Beginner mistakeTrying to read a shared value's .value directly in JSX during render — it must be read inside useAnimatedStyle or a worklet, not plain component code.
Intermediate mistakeAnimating layout properties (like width) frequently, which are more expensive than transform-based properties (translateX, scale) that don't trigger layout recalculation.

Composition & Interactions

Real interactions usually combine a gesture (drag, tap, long-press) with an animation that responds to it in real time — not just a fire-and-forget animation.

components/DraggableBox.tsx 
import Animated, { useSharedValue, useAnimatedStyle } from 'react-native-reanimated';
import { Gesture, GestureDetector } from 'react-native-gesture-handler';

export default function DraggableBox() {
  const translateX = useSharedValue(0);

  const pan = Gesture.Pan()
    .onChange((e) => {
      translateX.value += e.changeX;
    });

  const style = useAnimatedStyle(() => ({
    transform: [{ translateX: translateX.value }],
  }));

  return (
    <GestureDetector gesture={pan}>
      <Animated.View style={[{ width: 80, height: 80, backgroundColor: '#1f9d8a' }, style]} />
    </GestureDetector>
  );
}
  • Gesture.Pan() — describes a drag gesture. .onChange fires continuously as the finger moves, giving incremental deltas.
  • e.changeX — the change in horizontal position since the last event, added onto the running translateX shared value.
  • GestureDetector — connects the gesture definition to the actual view it should track touches on.
  • Because both the gesture handling and the style update happen via shared values, this drag tracks the finger with no perceptible lag, even under JS thread load.
Composing gestures Gesture Handler lets you combine gestures with Gesture.Simultaneous(), Gesture.Race(), or Gesture.Exclusive() — for example, a pinch-to-zoom that can happen at the same time as a pan, but a tap that should cancel a long-press.
Intermediate mistakeMixing the older PanResponder API with Gesture Handler in the same interaction, causing conflicting gesture recognition.
Professional mistakeNot resetting a shared value on gesture end (e.g., snapping back with withSpring), leaving dragged elements in an unintended final position.

Swipeable Components

Swipe-to-delete/archive is one of the most recognizable mobile interaction patterns — familiar from nearly every email and messaging app. Let's build the interaction, combining everything from this section.

components/SwipeableRow.tsx 
import Animated, { useSharedValue, useAnimatedStyle, withSpring } from 'react-native-reanimated';
import { Gesture, GestureDetector } from 'react-native-gesture-handler';

export default function SwipeableRow({ onDelete }: { onDelete: () => void }) {
  const translateX = useSharedValue(0);

  const pan = Gesture.Pan()
    .onChange((e) => { translateX.value += e.changeX; })
    .onEnd(() => {
      if (translateX.value < -100) {
        translateX.value = withSpring(-400);
        runOnJS(onDelete)();
      } else {
        translateX.value = withSpring(0); // snap back
      }
    });

  const style = useAnimatedStyle(() => ({
    transform: [{ translateX: translateX.value }],
  }));

  return (
    <GestureDetector gesture={pan}>
      <Animated.View style={style}>{/* row content */}</Animated.View>
    </GestureDetector>
  );
}
  • .onEnd() — fires once when the finger lifts; here it decides whether the swipe passed a "commit" threshold (-100) or should snap back.
  • runOnJS(onDelete)() — gesture callbacks run on the UI thread by default; runOnJS is required to safely call back into regular JS (like updating a list's state) from there.
  • withSpring — a physics-based animation that feels more natural for snap/settle motions than a fixed-duration withTiming.
Real-world example This exact pattern — pan gesture, threshold check, spring settle, runOnJS callback into a state update — is how swipe-to-archive works in most modern email and to-do apps.
Beginner mistakeCalling a state-setting function directly inside a gesture worklet without runOnJS, which throws, since worklets run in a different context than your React code.
Intermediate mistakeHardcoding the delete threshold in pixels rather than as a fraction of screen/row width, breaking on very different screen sizes.
Professional mistakeNot providing a non-gesture fallback (like a visible delete button) for accessibility — swipe-only interactions can be difficult for some users to discover or perform.

Section 6 final check

Why is runOnJS needed to call a regular JS function from a gesture callback?

Correct — worklets execute in a separate context for performance; runOnJS is the sanctioned way to call back into normal JavaScript/React code from there.
