Environment Variables

API keys, server URLs, and secrets should never be hardcoded directly into your source code — especially since your JS bundle can, with effort, be inspected by anyone who has your app installed. Environment variables keep configuration separate from code.

.env 
EXPO_PUBLIC_API_URL=https://api.myapp.com
EXPO_PUBLIC_ANALYTICS_KEY=abc123
app/index.tsx 
const apiUrl = process.env.EXPO_PUBLIC_API_URL;
fetch(apiUrl + '/posts');
  • The EXPO_PUBLIC_ prefix is required for any variable you want accessible in client-side app code — it's a deliberate, explicit opt-in so you can't accidentally expose a secret.
  • process.env.EXPO_PUBLIC_API_URL — Expo's build tooling replaces this at build time with the actual value from your .env file.
Critical distinction: public vs. private Anything prefixed EXPO_PUBLIC_ gets bundled into your client app and is technically extractable by anyone with the app installed — treat it like something printed on a public sign, never a secret. True secrets (payment provider secret keys, database credentials) belong only on your server, never in a variable your app bundle can read.
Type Example Where it lives
Public config API base URL, analytics key EXPO_PUBLIC_*, safe in the app bundle
Private secret Database password, Stripe secret key Server-only environment variables, never in the app
Beginner mistakePutting a genuinely secret API key behind an EXPO_PUBLIC_ variable, believing it's hidden because it's not visible in the source code you wrote.
Intermediate mistakeCommitting a real .env file with actual secrets to git — always .gitignore it and commit a .env.example template instead.

Quick check

Is a variable prefixed EXPO_PUBLIC_ safe to store a real secret API key in?

Correct — the prefix controls bundling, not secrecy. Anything bundled into the app can eventually be extracted.
TOPIC 2 / 4

Expo Router API Routes

Expo Router isn't only for screens — files named with a +api.ts suffix become real server endpoints, running on Node.js, deployed alongside your app. This means small backends can live in the very same project and repository as your app.

app/hello+api.ts 
export function GET(request: Request) {
  return Response.json({ message: 'Hello from the server!' });
}
  • app/hello+api.ts — the +api suffix marks this file as a server route, not a screen; it's automatically excluded from your client app bundle.
  • export function GET(request) — exporting a function named after an HTTP method (GET, POST, etc.) handles requests of that type at this route.
  • This route is reachable at /hello once your app is deployed — the client calls it exactly like any other API.
app/index.tsx (calling it from the client) 
const res = await fetch('/hello');
const data = await res.json();
Why this is genuinely useful, not just a toy A small endpoint that needs a private secret (like calling a payment provider's secret key) can't safely live in client code at all — an API route lets you keep that logic server-side without standing up a wholly separate backend project and deployment pipeline.
Beginner mistakePutting genuinely private secrets in a regular screen file instead of an API route — only server-side +api.ts code can safely use non-EXPO_PUBLIC_ secrets.
Intermediate mistakeBuilding an entire complex backend inside API routes when a dedicated backend service architecture would scale and organize better.
TOPIC 3 / 4

Streaming Data with Expo Fetch

Some responses shouldn't wait to fully complete before you start using them — think an AI chat response appearing word-by-word, rather than all at once after a long pause. Streaming lets you process a response incrementally as it arrives.

app/chat.tsx 
import { fetch } from 'expo/fetch';

async function streamReply(onChunk: (text: string) => void) {
  const response = await fetch('/chat-stream');
  const reader = response.body?.getReader();
  const decoder = new TextDecoder();

  while (true) {
    const { done, value } = await reader!.read();
    if (done) break;
    onChunk(decoder.decode(value));
  }
}
  • fetch from 'expo/fetch' — a fetch implementation with full streaming support, unlike React Native's traditional built-in fetch, which historically buffered the whole response.
  • response.body.getReader() — gives you a reader that yields chunks of the response as they arrive over the network, instead of waiting for completion.
  • TextDecoder — converts the raw binary chunks into readable text incrementally.
  • Each loop iteration calls onChunk as soon as new data is available — this is what powers a "typing" effect in a chat UI.
Real-world example This exact pattern is how AI chat apps render responses token-by-token as they're generated, rather than showing a loading spinner until the entire response is ready — meaningfully improving perceived responsiveness for long outputs.
Intermediate mistakeUsing the standard built-in fetch for a use case that needs true streaming — historically it doesn't expose a readable stream the same way.
Professional mistakeNot handling a dropped connection mid-stream, leaving the UI stuck mid-response with no error state or retry option.
TOPIC 4 / 4

Deploying with EAS Hosting

Once your project has API routes (Topic 2), something has to actually run that server code in production. EAS Hosting deploys your Expo Router project's server portion directly, without a separate backend deployment setup.

terminal 
# from your project root
eas deploy
  • eas deploy — builds your project's web/server output and deploys it to Expo's hosting infrastructure, giving you a live URL for your API routes.
  • Your mobile app's fetch calls to /hello or /chat-stream (from the previous two topics) resolve against this deployed URL in production, and your local dev server during development. 
Development:  app fetch("/hello")  ?  localhost dev server
Production:   app fetch("/hello")  ?  EAS Hosting deployment URL
When this replaces a separate backend, and when it doesn't For small apps — a handful of endpoints, some server-side secret handling, maybe a webhook receiver — EAS Hosting can be your entire backend. For larger systems with complex business logic, background jobs, or a large existing backend team, it's more common to use API routes for thin client-facing endpoints while a dedicated backend service (Section 11 covers Supabase as one option) handles the heavy lifting.
Beginner mistakeForgetting to update the client's base URL from a local development address to the deployed EAS Hosting URL before shipping to app stores.
Intermediate mistakeNot setting production environment variables/secrets on the hosting deployment itself, causing server-side code that worked locally to fail once deployed.
