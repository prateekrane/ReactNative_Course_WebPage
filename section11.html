Supabase UI Tour

Supabase is an open-source backend platform built on top of Postgres — a real, industry-standard relational database — with a dashboard that gives you instant tools for tables, authentication, file storage, and realtime data, without configuring any of it from scratch.

Why this matters compared to writing your own backend A real production backend with auth, a database, file storage, and realtime subscriptions normally takes days or weeks to wire up securely. Supabase gives you all four, pre-integrated, from the moment you create a project — letting Section 10's API routes stay thin, since the heavy lifting lives here.

TOPIC 2 / 3

Creating Tables

A table is a structured shape for your data — rows are records, columns are fields, and each column has a fixed type. Let's build one interactively before looking at the underlying SQL.

Try it: build a "todos" table Add a couple of columns below and watch the actual SQL and diagram update live — this is exactly what happens behind the scenes when you use the Table Editor's visual "add column" button.

Table builder

Generated SQL

CREATE TABLE todos (
  id uuid primary key default gen_random_uuid()
);

Reading the SQL yourself

SQL Editor 
create table todos (
  id uuid primary key default gen_random_uuid(),
  title text not null,
  is_complete bool default false,
  created_at timestamptz default now()
);
  • id uuid primary key default gen_random_uuid() — every table needs a unique identifier per row; a UUID auto-generated by the database is the standard, collision-safe default.
  • not null — a constraint: Postgres will reject any row that tries to insert this column as empty.
  • default false / default now() — automatic values applied when a row is created without explicitly specifying that column.
Row Level Security (RLS) — enabled by default New Supabase tables have RLS on by default, which blocks all access until you write a policy explicitly allowing it. This is a deliberate safety default: it's much safer to start locked-down and open specific access than to start open and forget to lock something down.
Beginner mistakeBeing confused why queries return no data or errors right after creating a table — it's very likely RLS is enabled with no policy yet, not a bug in your code.
Intermediate mistakeChoosing text for a numeric field "because it's simpler," losing the ability to sort/filter numerically at the database level.

Quick check

Why does a freshly created Supabase table often return no data, even though rows exist?

Correct — RLS defaults to "deny all" until you explicitly write a policy, which is a safety-first design choice.
TOPIC 3 / 3

Installation and CRUD Operations

CRUD stands for Create, Read, Update, Delete — the four basic operations every data-backed app needs. Let's connect the client library, then try all four live.

terminal 
npx expo install @supabase/supabase-js
lib/supabase.ts 
import { createClient } from '@supabase/supabase-js';

export const supabase = createClient(
  process.env.EXPO_PUBLIC_SUPABASE_URL!,
  process.env.EXPO_PUBLIC_SUPABASE_ANON_KEY!
);
  • The anon key is safe to expose in your client app (hence EXPO_PUBLIC_, from Section 10) — it identifies your project but relies on RLS policies to control what it can actually access.
C
Create

.insert() — adds a new row

R
Read

.select() — fetches rows

U
Update

.update() — modifies existing rows

D
Delete

.delete() — removes rows

app/todos.tsx 
// Create
await supabase.from('todos').insert({ title: 'Buy milk' });

// Read
const { data } = await supabase.from('todos').select('*');

// Update
await supabase.from('todos').update({ is_complete: true }).eq('id', todoId);

// Delete
await supabase.from('todos').delete().eq('id', todoId);
  • .from('todos') — targets the table created in the previous topic.
  • .eq('id', todoId) — a filter; without it, .update()/.delete() would apply to every row in the table, a common and dangerous mistake.

Beginner mistakeForgetting .eq(...) on an update() or delete() call — without a filter, the operation targets every row that RLS allows.
Intermediate mistakeNot checking the error field returned alongside data on every call — Supabase calls don't throw by default; errors come back as a value you must check.
Professional mistakeWriting overly permissive RLS policies ("allow all") during development and forgetting to tighten them before shipping to production.

Section 11 final check

What happens if you call .delete() on a table without an .eq(...) filter?

Correct — always double-check filters on update/delete calls before running them, especially against production data.
