Configuring EAS Update

EAS Update delivers over-the-air (OTA) updates — new JavaScript and assets — directly to installed apps, without an app store review. It cannot ship new native code (that still requires a full build), but the vast majority of everyday changes (UI tweaks, bug fixes, copy changes, new screens built from existing native modules) qualify.

The critical mental model: channels and branches A branch holds a sequence of updates (like a git branch). A channel is what a build actually listens to. You map a channel to a branch — this indirection is what lets you point production users at a different update stream than internal testers, without rebuilding the app.
app.json 
{
  "expo": {
    "updates": {
      "url": "https://u.expo.dev/your-project-id"
    },
    "runtimeVersion": { "policy": "appVersion" }
  }
}
  • runtimeVersion — a critical safety check: an update can only reach a build if their runtime versions match, preventing an OTA update built against newer native code from being sent to a build that doesn't have it yet.
  • The "appVersion" policy ties runtime version to your app's version string automatically — a common, safe default for most projects.
Why this indirection is worth the extra concept Without it, you'd need a hardcoded, one-to-one mapping of build to update stream. With channels, you can repoint which branch a channel serves at any time — for example, temporarily routing production to a hotfix branch — without needing a new build at all.
Beginner mistakeConfusing a channel and a branch as the same thing — a build's channel is fixed at build time; the branch it points to can change anytime.
Intermediate mistakeShipping a native-code-requiring change as if it were a simple OTA update — EAS Update will refuse or silently fail to apply changes needing new native modules.

Quick check

What's the difference between a branch and a channel in EAS Update?

Correct — this indirection is exactly what lets you retarget which updates a build receives without a new native build.

Deploying Updates

terminal 
eas update --branch production --message "Fix checkout crash"
  • --branch production — publishes this update to the production branch specifically.
  • --message — a human-readable note shown in your EAS dashboard's update history, invaluable when tracing which update caused an issue later.

Reverting Updates

Every published update is kept in your branch's history — reverting doesn't delete anything, it simply republishes an older update as the new "latest," so devices checking for updates receive the previous, known-good version.

Try a rollback yourself Below is a simulated update history for the production branch. Click "Revert to this" on an older entry to see exactly what a real rollback does.

What "revert" really does

Reverting publishes a new update entry that happens to contain the old version's code — the branch's history keeps growing forward, it never gets rewritten. This is why rollbacks are safe and instant.

terminal 
# list recent updates on a branch to find the one to restore
eas update:list --branch production

# republish a specific earlier update as the new latest
eas update:republish --group <update-group-id>
  • eas update:list — shows every published update on a branch with its ID, message, and timestamp — exactly the history simulated above.
  • eas update:republish — doesn't create new code, it re-points "latest" back at a specific previous update group.
Rollbacks vs. a fresh fix-forward update A rollback is fast and safe when the previous version was known-good. If the bug existed for a while and reverting would also remove other since-shipped fixes users need, publishing a new, forward-fixing update is usually the better call instead.
Beginner mistakePanicking and trying to "delete" a bad update — EAS Update's model is append-only; you always publish forward, even when reverting.
Intermediate mistakeReverting a runtimeVersion-incompatible update onto a build that has since moved to a newer runtime version, which won't apply for the same safety reasons covered in Topic 1.

Section 14 final check

What actually happens when you "revert" an EAS Update?

Correct — the branch history is append-only; reverting is really "publish forward with old content," not a deletion.
