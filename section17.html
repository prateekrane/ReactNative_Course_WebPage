Payment Basics

Before writing any payment code, the single most important question is: what are you selling? Apple and Google's store rules — not just convenience — dictate which payment system you're legally required to use.

The rule that decides everything If you're selling digital content or features consumed inside the app (premium features, subscriptions, virtual currency), Apple and Google require you to use their own in-app purchase systems — you are not allowed to use Stripe directly for this, even if you wanted to. Physical goods, services consumed outside the app (a ride, a hotel booking), and donations are typically exempt.

In-app Purchases

RevenueCat is the standard way to implement in-app purchases in React Native — it wraps Apple's StoreKit and Google's Billing Library behind one consistent API, and handles the genuinely tricky part: reliably verifying that a purchase actually happened.

app/_layout.tsx 
import Purchases from 'react-native-purchases';
import { useEffect } from 'react';

export default function RootLayout() {
  useEffect(() => {
    Purchases.configure({ apiKey: process.env.EXPO_PUBLIC_REVENUECAT_KEY! });
  }, []);
  // ...rest of layout
}
app/paywall.tsx 
async function handleSubscribe() {
  const offerings = await Purchases.getOfferings();
  const pkg = offerings.current!.availablePackages[0];

  try {
    const { customerInfo } = await Purchases.purchasePackage(pkg);
    if (customerInfo.entitlements.active['premium']) {
      // unlock premium features
    }
  } catch (e: any) {
    if (!e.userCancelled) { /* show a real error */ }
  }
}
  • getOfferings() — fetches the subscription/product options you've configured in the RevenueCat dashboard, which mirror what you set up in App Store Connect / Play Console.
  • purchasePackage(pkg) — triggers the native purchase sheet (Face ID / fingerprint confirmation) and, on success, returns verified customer info.
  • entitlements.active['premium'] — the key check: RevenueCat's verified record of what this user is currently entitled to, regardless of which platform they purchased on.
  • e.userCancelled — distinguishing a user backing out of the purchase sheet from a genuine error is important; don't show a scary error message for a simple cancellation.

Section 17 final check

Why check customerInfo.entitlements instead of a local "isPremium" flag your own code sets?

Correct — a locally-set flag can be manipulated or become out of sync; the verified entitlement is the source of truth.
