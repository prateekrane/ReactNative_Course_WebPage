Expo Notifications

Every push notification system works the same basic way: your device registers with the OS's push service and receives a unique token, you send that token to your server, and your server later sends notifications to a push service using that token.

hooks/usePushToken.ts 
import * as Notifications from 'expo-notifications';
import { useEffect, useState } from 'react';

export function usePushToken() {
  const [token, setToken] = useState<string | null>(null);

  useEffect(() => {
    (async () => {
      const { status } = await Notifications.requestPermissionsAsync();
      if (status !== 'granted') return;

      const { data } = await Notifications.getExpoPushTokenAsync();
      setToken(data);
      // send `data` to your backend here to store against the current user
    })();
  }, []);

  return token;
}
  • requestPermissionsAsync() — shows the OS's native permission prompt; without explicit permission, no notification can ever be delivered.
  • getExpoPushTokenAsync() — returns a token in the format ExponentPushToken[...], which Expo's push service uses to route notifications to this specific device.
  • The token must be sent to and stored by your backend — Expo's service doesn't know which "user" a token belongs to, only your server does.
server: sending a notification 
await fetch('https://exp.host/--/api/v2/push/send', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    to: pushToken,
    title: 'New message',
    body: 'You have a new message from Jane',
  }),
});
Why "Expo's push service" and not APNs/FCM directly? Apple's APNs and Google's FCM each have their own separate, more complex authentication setup. Expo's push service acts as a unified layer in front of both — your server sends one consistent request format, and Expo handles routing to the correct platform-specific service.
Beginner mistakeForgetting to request permission before trying to fetch a push token — the request will simply fail or return nothing.
Intermediate mistakeNever updating a stored token when it changes (tokens can rotate) — leading to notifications silently failing for some users over time.
Professional mistakeNot handling the case where a user denies permission gracefully — the app should function normally, just without notifications, not break.

Quick check

What must happen before a device can receive any push notification?

Correct — without permission and a valid, server-stored token, there's no way to route a notification to that device.

OneSignal Integration

Expo's push service is great for getting started, but production apps with marketing/engagement needs often reach for OneSignal — a dedicated push notification platform with segmentation, scheduling, analytics, and A/B testing built in.

Expo Notifications OneSignal
Cost Free Free tier, paid tiers for scale/features
Setup complexity Minimal Moderate (dashboard + SDK setup)
Audience segmentation You build it yourself Built-in dashboard tooling
Scheduling/campaigns You build it yourself Built-in
Analytics Minimal Detailed delivery/open analytics
app/_layout.tsx 
import { OneSignal } from 'react-native-onesignal';
import { useEffect } from 'react';

export default function RootLayout() {
  useEffect(() => {
    OneSignal.initialize('YOUR_ONESIGNAL_APP_ID');
    OneSignal.Notifications.requestPermission(true);
  }, []);

  // ...rest of layout
}
  • OneSignal.initialize(...) — connects your app to your OneSignal dashboard project using its App ID, found in your OneSignal account settings.
  • Notifications.requestPermission(true) — triggers the native permission prompt, same underlying OS mechanism as Expo Notifications.
Deciding which to use Start with Expo Notifications for MVPs and simple transactional notifications (order updates, chat messages). Move to OneSignal once you need marketing campaigns, audience segments, or detailed delivery analytics that would otherwise require building your own dashboard.
Beginner mistakeIntegrating a third-party push SDK like OneSignal directly inside Expo Go — like other custom native modules (Section 2), this requires a development build.
Intermediate mistakeRunning both Expo Notifications and OneSignal simultaneously without a clear reason, causing duplicate permission prompts and confusing token management.

Rich Notifications iOS

A plain notification shows title and body text. A rich notification can include an image, video thumbnail, or custom UI — like a message preview showing the actual photo someone sent.

server: sending a rich notification 
await fetch('https://exp.host/--/api/v2/push/send', {
  method: 'POST',
  headers: { 'Content-Type': 'application/json' },
  body: JSON.stringify({
    to: pushToken,
    title: 'Jane sent a photo',
    body: 'Tap to view',
    mutableContent: true,
    // custom data your Notification Service Extension reads to fetch the image
    data: { imageUrl: 'https://example.com/photo.jpg' },
  }),
});
  • mutableContent: true — tells iOS this notification is allowed to be modified before display, which is required for attaching media.
  • Actually attaching the image requires a small native Notification Service Extension — a separate, tiny native target that downloads the image and attaches it before the notification is shown. This is native iOS configuration, set up once per project.
Why this needs native configuration, not just a JS flag iOS delivers notifications to your app even when it's fully closed, via a special background extension process — this is lower-level than your main React Native JS bundle, which is why media attachments require native Xcode project setup (usually via an Expo config plugin) rather than pure JS code.
Intermediate mistakeSetting mutableContent: true without the accompanying Notification Service Extension configured — the flag alone does nothing without the native piece.
Professional mistakeNot handling image download failures inside the Service Extension, causing the entire notification to fail silently instead of gracefully falling back to text-only.

Section 9 final check

What's required to show an image inside an iOS push notification?

Correct — rich media notifications require both the mutable content flag and a small native extension to actually process the attachment.
