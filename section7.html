What is Expo UI?

Every component you've used so far (View, Text, Pressable) is a React Native abstraction that maps to a native view, but it's still React Native's own rendering layer. Expo UI is different: it renders using the platform's actual, first-party UI toolkit — SwiftUI on iOS, Jetpack Compose on Android — directly.

Why this exists Apple and Google each ship rapidly evolving native design systems (Liquid Glass, Material You) with animations, accessibility behavior, and platform conventions that are extremely hard to replicate pixel-for-pixel in a cross-platform abstraction. Expo UI lets you use the real thing where it matters most, while keeping your app logic in React.

The mental model: SwiftUI and Compose

Both SwiftUI and Jetpack Compose are declarative UI frameworks — meaning you describe what the UI should look like for a given state, not the imperative steps to update it. If this sounds familiar, it's because React itself works the same way. This is why Expo UI feels conceptually consistent with React Native rather than bolted-on.

Concept React SwiftUI Jetpack Compose
Unit of UI Component View Composable function
Reactivity useState @State remember { mutableStateOf() }
Styling style objects modifiers modifiers
When to reach for Expo UI vs. standard components Standard View/Text/Pressable remain the right default for most screens, especially anything cross-platform-consistent (like a custom-branded checkout flow). Expo UI shines for platform-native elements users already have muscle memory for — native pickers, native forms, native list styles, context menus.
Beginner mistakeAssuming Expo UI replaces React Native's core components entirely — it's a complementary tool for specific native-feeling UI, not a wholesale replacement.
Intermediate mistakeUsing Expo UI for a component that must look pixel-identical on both platforms — by design, SwiftUI and Compose renderings can differ slightly, which is often desirable but sometimes isn't.

TOPIC 2 / 8

Expo UI on iOS

On iOS, Expo UI exposes SwiftUI components through a JS API — you write JSX, but under the hood it constructs real SwiftUI views.

app/settings.tsx 
import { Host, Form, Section, Switch } from '@expo/ui/swift-ui';
import { useState } from 'react';

export default function SettingsScreen() {
  const [notifsOn, setNotifsOn] = useState(true);

  return (
    <Host style={{ flex: 1 }}>
      <Form>
        <Section title="Notifications">
          <Switch value={notifsOn} onValueChange={setNotifsOn} label="Push Notifications" />
        </Section>
      </Form>
    </Host>
  );
}
  • Host — the bridge component that mounts a SwiftUI view hierarchy inside your React Native tree; everything inside it renders via SwiftUI, not RN's own renderer.
  • Form / Section — map directly to SwiftUI's own grouped-list form components, giving you the exact native Settings-app look for free.
  • State (notifsOn) still lives in React — Expo UI components accept props and fire callbacks exactly like normal React Native components.
Real-world example A Settings screen built this way looks and behaves identically to Apple's own Settings app — correct native toggle animation, correct grouped list styling, correct dynamic type support — without hand-rebuilding any of that visually in React Native's styling system.
Intermediate mistakeMixing regular RN components and SwiftUI components without a clear boundary, making layout debugging confusing across two different rendering systems.

TOPIC 3 / 8

Expo UI on Android

The Android equivalent renders real Jetpack Compose composables, using the same JSX-first API shape for consistency.

app/settings.tsx 
import { Host, Column, Switch, Text } from '@expo/ui/jetpack-compose';
import { useState } from 'react';

export default function SettingsScreen() {
  const [notifsOn, setNotifsOn] = useState(true);

  return (
    <Host style={{ flex: 1 }}>
      <Column>
        <Text>Push Notifications</Text>
        <Switch value={notifsOn} onValueChange={setNotifsOn} />
      </Column>
    </Host>
  );
}
  • @expo/ui/jetpack-compose — the Android-specific package; note this is a different import than the iOS example, since they wrap genuinely different native toolkits.
  • Column — Compose's vertical layout container, analogous to a View with flexDirection: 'column', but rendered by Compose itself.
Writing cross-platform Expo UI screens Since the two imports differ, real apps typically use Platform.select() or separate .ios.tsx/.android.tsx files to swap between them, sharing the surrounding screen logic and state.
Beginner mistakeImporting the SwiftUI package on Android (or vice versa) — these are platform-specific native bindings, not universal components.

Quick check

Why are the SwiftUI and Jetpack Compose Expo UI imports separate packages?

Correct — SwiftUI is iOS-only and Compose is Android-only at the native level, so Expo UI mirrors that split rather than pretending they're interchangeable.

TOPIC 4 / 8

Native state and worklets

Some Expo UI interactions need to update at native speed — a slider being dragged, a live preview responding instantly — without a round trip through React state on every frame. This is where worklets (the same underlying concept from Reanimated, Section 6) come back into play.

components/VolumeSlider.tsx 
import { Host, Slider } from '@expo/ui/swift-ui';
import { useSharedValue } from 'react-native-reanimated';

export default function VolumeSlider() {
  const volume = useSharedValue(0.5);

  return (
    <Host>
      <Slider
        defaultValue={0.5}
        onValueChange={(v) => {
          'worklet';
          volume.value = v; // updates natively, no JS round trip per frame
        }}
      />
    </Host>
  );
}
  • 'worklet' directive — tells the Reanimated/JSI system this function should run on the native/UI thread instead of the regular JS thread.
  • Writing to a shared value inside a worklet avoids triggering a React re-render for every tiny slider movement — only components actually reading that shared value (via useAnimatedStyle) update, and they do so on the native side.
Why this matters specifically for Expo UI Native components can fire events extremely frequently (drag, scroll, live camera data). Routing every single one through React state would be needlessly expensive; worklets let native UI and native state stay in sync directly, only touching React when something meaningfully needs to update.
Intermediate mistakeCalling regular React state setters directly inside a worklet without runOnJS, causing errors — the same rule from Section 6 applies here.
Professional mistakeOverusing worklets for interactions that don't actually need frame-perfect native updates, adding unnecessary complexity.

TOPIC 5 / 8

Universal components

Not every Expo UI component needs a separate iOS/Android implementation in your code — some are exposed as universal components: one import, one API, that internally renders via SwiftUI on iOS and Compose on Android without you branching manually.

components/ThemedButton.tsx 
import { Button } from '@expo/ui'; // universal, platform-aware entry point

<Button text="Continue" onPress={() => {}} />
// Renders as a real SwiftUI Button on iOS,
// a real Compose Button on Android — same code, same props.
Universal vs. platform-specific: how to choose Start with universal components whenever your design accepts each platform's native look for that element (buttons, switches, sliders). Drop to the platform-specific package only when you need a capability or visual only one platform's toolkit exposes.
Beginner mistakeManually writing Platform.OS === 'ios' branches to pick between SwiftUI/Compose imports for components that already have a universal version — extra complexity with no benefit.
  • Beginner Replace a Pressable-based custom button with a universal Expo UI Button and compare how it looks on both platforms (in documentation screenshots, if you don't have both devices).

TOPIC 6 / 8

Custom SwiftUI views and modifiers

When Expo UI's built-in components aren't enough, you can write actual Swift code defining a custom SwiftUI view, and expose it to your React Native app as a component — this is your first real taste of the Native Modules concepts from Section 18, in a UI-focused form.

modules/gradient-card/ios/GradientCardView.swift 
import SwiftUI
import ExpoModulesCore

struct GradientCardView: View, Expo.Recordable {
  var title: String

  var body: some View {
    Text(title)
      .padding()
      .background(
        LinearGradient(colors: [.orange, .pink], startPoint: .leading, endPoint: .trailing)
      )
      .cornerRadius(12)
  }
}
  • This file lives inside a small custom Expo Module (Section 18 covers the full module structure) — Swift code, compiled as part of your native build.
  • .background(LinearGradient(...)) and .cornerRadius(12) are SwiftUI modifiers — chained calls that transform a view's appearance, conceptually similar to chaining style objects, but applied as method calls in Swift.
app/index.tsx 
import { GradientCardView } from 'gradient-card';

<GradientCardView title="Premium" style={{ height: 80 }} />
Why go this deep? Some visual effects (specific gradients, blurs, native animations tied to system settings) are far easier — and more correct — to express directly in SwiftUI than to approximate in React Native's styling system. This path exists for exactly those cases, not as a default workflow.
Professional mistakeReaching for a custom SwiftUI view before checking whether a universal or platform-specific Expo UI component already covers the need.

TOPIC 7 / 8

Custom Compose views and modifiers

The Android mirror of the previous topic: a custom Jetpack Compose composable, exposed to React Native the same way.

modules/gradient-card/android/GradientCardView.kt 
import androidx.compose.foundation.background
import androidx.compose.foundation.layout.padding
import androidx.compose.foundation.shape.RoundedCornerShape
import androidx.compose.material3.Text
import androidx.compose.ui.Modifier
import androidx.compose.ui.graphics.Brush
import androidx.compose.ui.unit.dp

@Composable
fun GradientCardView(title: String) {
  Text(
    text = title,
    modifier = Modifier
      .padding(12.dp)
      .background(
        Brush.horizontalGradient(listOf(OrangeColor, PinkColor)),
        RoundedCornerShape(12.dp)
      )
  )
}
  • @Composable — marks this function as a Compose UI-building function, Compose's equivalent of a component.
  • Modifier.padding(...).background(...) — Compose's modifier chain, conceptually identical to SwiftUI's modifier chain from the previous topic, just Kotlin syntax instead of Swift.
Notice the pattern across both platforms Same overall shape: a small native UI function, decorated with chained style modifiers, exposed to JS through an Expo Module. Once you've done it once on one platform, the second platform is mostly a syntax translation, not a new mental model.
Intermediate mistakeForgetting that Compose modifier order matters — padding then background produces a different visual result than background then padding.

Quick check

What is the Jetpack Compose equivalent of a SwiftUI "modifier"?

Correct — both frameworks use a chainable modifier concept to layer styling and behavior onto a view/composable.

TOPIC 8 / 8

Architecting a maintainable screen

Mixing standard React Native components, universal Expo UI components, and fully custom native views on one screen can get messy fast without a clear structure. Here's a pattern that keeps it maintainable. 

ProfileScreen (RN layout: View, ScrollView)
+-- ProfileHeader        — standard RN components
+-- NativeSettingsForm    — Expo UI universal Form/Section
+-- GradientBadge        — fully custom SwiftUI/Compose view

The guiding principle

  • Layout stays in React Native. Use View/ScrollView/Flexbox for overall screen structure — this keeps layout logic in one consistent system.
  • Reach for Expo UI at the leaf level. Individual native-feeling controls (forms, pickers, switches) are good candidates; the surrounding scaffolding usually shouldn't be.
  • Isolate fully custom native views behind a clear component boundary (like GradientBadge above), so the rest of the screen doesn't need to know or care that it's backed by Swift/Kotlin.
Why this ordering matters If custom native views are scattered without a clear boundary, debugging layout issues means constantly context-switching between React Native's box model and each native toolkit's own layout system. Isolating them keeps that complexity contained and predictable.
Intermediate mistakeNesting Expo UI's Host components deeply and inconsistently throughout a screen instead of at clear, intentional boundaries.
Professional mistakeNot documenting which parts of a screen are backed by native code for teammates unfamiliar with Swift/Kotlin, causing confusion during unrelated bug fixes.

Section 7 final check

What should typically stay in React Native's own layout system, even on an Expo UI—heavy screen?

Correct — keeping structural layout in RN's own system while using Expo UI at the leaf/control level keeps the mental model manageable.
