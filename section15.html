Getting Started with Workflows

Everything you've run manually so far — eas build, eas submit, eas update, Maestro tests — can be triggered automatically by EAS Workflows: YAML files that define jobs, run on triggers like a git push, executed on Expo's own CI infrastructure.

.eas/workflows/on-push-main.yml 
name: Build on push to main

on:
  push:
    branches: ['main']

jobs:
  build_ios:
    type: build
    params:
      platform: ios
      profile: preview
  • on.push.branches — the trigger; this workflow only runs when commits land on main, not every branch.
  • jobs.build_ios — a named job; you'll typically have several (build, test, submit) that can depend on each other's results.
  • type: build — tells EAS this job should run an actual build, using the parameters below it — the same underlying build you triggered manually with eas build in Section 12.

Why this matters, beyond convenience Manual commands are easy to forget or run inconsistently — one engineer's local build profile flag typo shouldn't affect what ships. Codifying the exact command and parameters in a workflow file means every trigger runs identically, and the file itself is reviewable, versioned code your whole team can see and improve.
Beginner mistakeWriting a trigger with no branch filter, causing an expensive build job to fire on every single push to every branch.
Intermediate mistakeDuplicating near-identical workflow files for iOS and Android instead of parameterizing platform within jobs of one workflow.

Quick check

What determines when an EAS Workflow actually runs?

Correct — the on block defines exactly what event causes the workflow to run automatically.

E2E with Maestro Workflow

Recall Maestro from Section 8 — running E2E flows manually on your own machine. In CI, a workflow job builds a fresh app and runs your Maestro flows against it automatically, catching regressions before they ever reach a teammate or reviewer.

.eas/workflows/e2e-tests.yml 
name: E2E tests on pull request

on:
  pull_request: {}

jobs:
  build:
    type: build
    params: { platform: ios, profile: preview }

  maestro:
    needs: [build]
    type: maestro
    params:
      build_id: \${{ needs.build.outputs.build_id }}
      flow_path: .maestro/login-flow.yaml
  • needs: [build] — this job waits for build to finish successfully before starting; workflows can express dependencies between jobs, not just run everything at once.
  • build_id: ...needs.build.outputs.build_id — passes the exact freshly-built artifact from the previous job into Maestro, so the E2E test always runs against the code from this specific pull request.
  • flow_path — points at the same YAML flow file format you wrote by hand in Section 8.

Real-world example A team can require this exact workflow to pass before a pull request is mergeable — meaning a broken login flow is caught automatically on every PR, not discovered after merging to main or, worse, after release.
Beginner mistakeForgetting needs: [build], causing the Maestro job to try running against a build that doesn't exist yet.
Intermediate mistakeRunning the full E2E suite on every single commit instead of on pull requests, dramatically slowing down everyday development feedback loops.

Build and Submit Workflow

The final piece: chaining a production build directly into store submission, for both platforms, triggered by something as simple as pushing a git tag.

.eas/workflows/release.yml 
name: Release to both stores

on:
  push:
    tags: ['v*']

jobs:
  build_ios:
    type: build
    params: { platform: ios, profile: production }

  build_android:
    type: build
    params: { platform: android, profile: production }

  submit_ios:
    needs: [build_ios]
    type: submit
    params: { build_id: \${{ needs.build_ios.outputs.build_id }} }

  submit_android:
    needs: [build_android]
    type: submit
    params: { build_id: \${{ needs.build_android.outputs.build_id }} }
  • tags: ['v*'] — triggers only when a version tag like v2.3.0 is pushed, a common convention for marking an intentional release commit.
  • build_ios and build_android have no needs between them — they run in parallel automatically, since neither depends on the other.
  • Each submit_* job depends only on its matching platform's build — an iOS build failure doesn't block the Android submission from proceeding.

Intermediate mistakeAccidentally creating a dependency between the two platforms' chains, making an iOS delay unnecessarily block the Android release.
Professional mistakeNot gating the tag-push trigger behind any manual approval step for production releases, allowing an accidental tag push to trigger a real store submission.

Section 15 final check

In the release workflow above, why don't build_ios and build_android need a needs relationship between them?

Correct — jobs only run sequentially when a real data/ordering dependency exists between them, expressed via needs.
