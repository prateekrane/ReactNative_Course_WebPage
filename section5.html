File-Based Routing

Recall from Section 1: every file inside app/ becomes a route automatically. Let's formalize the rules. 

app/
+-- index.tsx           ?  "/"
+-- about.tsx            ?  "/about"
+-- settings/
│   +-- index.tsx        ?  "/settings"
│   +-- notifications.tsx ?  "/settings/notifications"
+-- _layout.tsx             (wraps every screen — not a route itself)
File name pattern Meaning
index.tsx The default route for that folder
_layout.tsx Shared UI/config wrapping all sibling and child routes — never a route itself
[id].tsx A dynamic segment — covered next topic
(group)/ A folder in parentheses groups routes without adding to the URL — used for organization
Why this is powerful You can understand an unfamiliar app's entire navigation structure just by scanning its app/ folder — no separate navigation config file to keep in sync with reality.
Beginner mistakeNaming a component file that shouldn't be a screen (like a shared component) directly inside app/, accidentally creating an unwanted route.
Intermediate mistakeForgetting that _layout.tsx is required in a folder for shared elements (like a header) to wrap that folder's screens.

TOPIC 2 / 8

Dynamic Routes

Most real screens aren't static — a product page needs a product ID, a profile page needs a username. Square brackets in a filename create a dynamic segment.

app/product/[id].tsx 
import { useLocalSearchParams } from 'expo-router';
import { Text } from 'react-native';

export default function ProductScreen() {
  const { id } = useLocalSearchParams<{ id: string }>();
  return <Text>Product #{id}</Text>;
}
  • app/product/[id].tsx — the brackets tell Expo Router this segment is variable; navigating to /product/42 or /product/anything both match this file.
  • useLocalSearchParams() — reads whatever value was in that dynamic segment for the current navigation.
app/index.tsx 
<Link href={`/product/\${product.id}`}>{product.name}</Link>
Catch-all routes [...slug].tsx (with three dots) matches any number of remaining path segments — useful for things like a CMS-driven page tree where depth isn't fixed in advance.
Beginner mistakeForgetting params always arrive as strings — id is "42", not the number 42, until you convert it.
Intermediate mistakeUsing useGlobalSearchParams when useLocalSearchParams was intended, picking up stale params from a parent route unexpectedly.

Quick check

What file name would correctly handle the route /user/[username]?

Correct — square brackets are Expo Router's syntax for a dynamic segment.

TOPIC 3 / 8

Stack & Tabs

Two navigation patterns cover the vast majority of apps: a Stack (screens pushed on top of each other, with a back button) and Tabs (persistent bottom navigation between top-level sections).

app/(tabs)/_layout.tsx 
import { Tabs } from 'expo-router';

export default function TabLayout() {
  return (
    <Tabs>
      <Tabs.Screen name="index" options={{ title: 'Home' }} />
      <Tabs.Screen name="search" options={{ title: 'Search' }} />
      <Tabs.Screen name="profile" options={{ title: 'Profile' }} />
    </Tabs>
  );
}
  • (tabs) — a parenthesized folder name; it groups these three screens under a shared tab bar without adding "/tabs" to any URL.
  • Tabs.Screen name="index" — name must match the actual filename (without extension) inside that folder.
They combine, constantly A typical app nests these: a root Stack contains a Tabs layout as one screen, and tapping into a list item from within a tab pushes a new screen onto the Stack, temporarily hiding the tab bar. This nesting is normal and expected.
Beginner mistakeNaming a tab route folder tabs/ instead of (tabs)/ — without parentheses, "tabs" becomes part of the URL, which usually isn't intended.
Intermediate mistakePutting every screen in one flat Stack instead of grouping related screens, making the navigation structure hard to reason about as the app grows.

TOPIC 4 / 8

Router Hooks

Beyond the Link component, Expo Router provides hooks for navigating and reading route state imperatively from your logic.

Hook Purpose
useRouter() Gives you router.push(), router.replace(), router.back() for navigating from event handlers
useLocalSearchParams() Reads dynamic segment values and query params for the current screen
usePathname() Returns the current URL path as a string
useSegments() Returns the current route as an array of path segments — useful for conditional layout logic
app/login.tsx 
import { useRouter } from 'expo-router';

export default function Login() {
  const router = useRouter();

  function handleLoginSuccess() {
    router.replace('/home'); // replace, not push — see note below
  }
  // ...
}
push vs replace push adds a new screen on top, leaving the previous one reachable via back. replace swaps the current screen out entirely — used after login so the user can't tap "back" and land on the login screen again.
Beginner mistakeUsing push after a login/onboarding flow, letting users navigate "back" into a screen they shouldn't revisit.
Intermediate mistakeCalling router methods during render instead of inside an event handler or effect, causing navigation to fire at the wrong time.

Quick check

After a successful login, which navigation method usually makes more sense?

Correct — replace removes the login screen from history so the back button can't return to it.

TOPIC 5 / 8

Authentication Flow

Almost every real app needs to show different screens to logged-in vs. logged-out users, and prevent direct navigation into protected screens. Expo Router handles this with route groups and a redirect check in a root layout. 

app/
+-- (auth)/
│   +-- login.tsx        ?  "/login"
│   +-- signup.tsx       ?  "/signup"
+-- (app)/
│   +-- index.tsx        ?  "/"  (protected)
│   +-- profile.tsx      ?  "/profile" (protected)
+-- _layout.tsx              (checks auth, redirects)
app/_layout.tsx 
import { Slot, Redirect } from 'expo-router';
import { useAuth } from '../hooks/useAuth';

export default function RootLayout() {
  const { isLoggedIn, isLoading } = useAuth();

  if (isLoading) return null; // or a splash screen
  if (!isLoggedIn) return <Redirect href="/login" />;

  return <Slot />; // renders whichever child route matched
}
  • <Redirect href="/login" /> — declaratively redirects, rather than imperatively calling router.replace in an effect — cleaner and avoids a flash of the wrong screen.
  • <Slot /> — renders whatever the currently matched child route is; used in layouts that don't need a Stack or Tabs wrapper themselves.
  • The isLoading check matters: without it, you'd briefly redirect to /login even for already-logged-in users while the auth check is still in flight.
Real-world example A banking app keeps all account screens under an (app) group protected by exactly this pattern — attempting to deep-link directly into /accounts/1234 without a valid session redirects straight to login, no protected data ever renders.
Beginner mistakeOnly hiding a "Login" link in the UI, without an actual layout-level check — a user can still navigate directly to the protected URL.
Intermediate mistakeNot handling the loading state, causing a visible flash of the login screen for already-authenticated users on app start.
Professional mistakeStoring auth tokens insecurely (e.g. in plain AsyncStorage) instead of a secure storage API designed for sensitive data.
  • Intermediate Build a minimal useAuth hook backed by a boolean state, and wire up the redirect logic shown above.

TOPIC 6 / 8

Role-Based Access

Beyond "logged in or not," many apps have roles — an admin sees different screens than a regular user. This builds directly on the authentication pattern above.

app/(app)/admin/_layout.tsx 
import { Slot, Redirect } from 'expo-router';
import { useAuth } from '../../../hooks/useAuth';

export default function AdminLayout() {
  const { user } = useAuth();

  if (user?.role !== 'admin') {
    return <Redirect href="/" />;
  }

  return <Slot />;
}
  • Placing this check in admin/_layout.tsx protects every screen inside that folder automatically — no need to repeat the check in each individual admin screen.
  • user?.role !== 'admin' — the optional chaining (?.) safely handles the case where user is still null.
Client-side checks are UX, not security This pattern controls what a legitimate user sees. Your backend API must independently verify role permissions on every request — a client-side redirect can always be bypassed by a determined attacker calling your API directly.
Beginner mistakeTreating a client-side role check as sufficient security, without matching enforcement on the backend.
Intermediate mistakeDuplicating the role check in every screen file instead of centralizing it in a shared layout for that section.

TOPIC 7 / 8

Deep Linking

Deep linking lets an external source — a push notification, a text message, a website — open your app directly to a specific screen, instead of just the home screen.

The good news with Expo Router Because your file structure already defines URL-like routes, deep linking mostly comes for free — myapp://product/42 maps to app/product/[id].tsx automatically, the same route your in-app Link components already use.
app.json 
{
  "expo": {
    "scheme": "myapp",
    "ios": { "associatedDomains": ["applinks:myapp.com"] },
    "android": {
      "intentFilters": [{
        "action": "VIEW",
        "data": [{ "scheme": "https", "host": "myapp.com" }],
        "category": ["BROWSABLE", "DEFAULT"]
      }]
    }
  }
}
  • "scheme": "myapp" — registers myapp:// as a custom URL scheme your app responds to.
  • associatedDomains / intentFilters — configure "Universal Links" (iOS) and "App Links" (Android), letting regular https://myapp.com/product/42 links open your app directly instead of a browser, when installed.
Link type Example Fallback if app not installed
Custom scheme myapp://product/42 Fails silently — only works if the app is installed
Universal/App Link https://myapp.com/product/42 Opens in browser normally
Beginner mistakeTesting deep links only via a custom scheme, then being surprised universal links behave differently and need separate platform configuration.
Intermediate mistakeForgetting that a fresh app install needs to fully register its intent filters/associated domains — testing a build that predates a scheme change won't reflect the update.

TOPIC 8 / 8

Liquid Glass Bottom Tabs

Recent iOS versions introduced a "Liquid Glass" translucent, blurred material design language for system UI elements, including tab bars. Expo Router's Tabs component supports this native visual style on supported platforms.

app/(tabs)/_layout.tsx 
import { Tabs } from 'expo-router';
import { BlurView } from 'expo-blur';

export default function TabLayout() {
  return (
    <Tabs
      screenOptions={{
        tabBarBackground: () => <BlurView intensity={80} style={{ flex: 1 }} />,
        tabBarStyle: { position: 'absolute' }, // lets content show through
      }}
    >
      <Tabs.Screen name="index" options={{ title: 'Home' }} />
    </Tabs>
  );
}
  • tabBarBackground — lets you render a custom component (here, a blur effect) behind the tab bar's icons/labels.
  • position: 'absolute' on tabBarStyle — makes the tab bar float over content rather than push it up, which is necessary for the translucent "glass" effect to show content scrolling underneath.
Design consideration, not just code Because the tab bar is now translucent and floats over content, your scrollable content needs bottom padding/inset so the last items aren't visually hidden behind it — check useBottomTabBarHeight() for the exact value to pad by.
Intermediate mistakeForgetting bottom-content padding after switching to a floating/glass tab bar, causing the last list item to sit behind the tab bar.
Professional mistakeApplying platform-specific visual treatments without a graceful fallback on platforms/OS versions that don't support the effect.

Section 5 final check

What must you account for when a tab bar becomes translucent and floats over content?

Correct — floating UI elements need corresponding content padding, or the bottom-most content becomes partially obscured.
