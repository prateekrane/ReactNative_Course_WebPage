Designing with Figma

Most professional React Native apps start life as a Figma design, not code. Being able to read a Figma file and translate it accurately into components and styles is a core, underrated developer skill.

What to look for in a Figma file

??
Spacing values

Select any frame in Figma and its right panel shows exact padding/gap in pixels — copy these directly into your styles.

??
Color styles

Well-organized files define named color styles (e.g. "Primary/500") rather than raw hex values scattered everywhere.

Aa
Text styles

Font family, size, weight, and line-height are all inspectable per text layer — these map directly to your Text styles.

??
Auto Layout

Figma's "Auto Layout" feature maps almost one-to-one onto Flexbox — direction, gap, and alignment translate directly.

The mental model Figma's Auto Layout is Flexbox, just with different names: "Auto Layout direction: Vertical" is flexDirection: 'column'. "Spacing between items" is gap. Learning to read one teaches you the other.
Figma propertyReact Native equivalent
Auto Layout: HorizontalflexDirection: 'row'
Spacing between itemsgap
Padding (all sides shown)padding / paddingHorizontal / paddingVertical
Corner radiusborderRadius
Fill colorbackgroundColor

A practical workflow

  1. Identify repeated patterns first (buttons, cards, list items) — build these as components before the full screen.
  2. Work outside-in: get the overall screen layout (Flexbox structure) right before polishing individual pieces.
  3. Extract a small design-tokens file (colors, spacing, font sizes) from the Figma styles before writing screen code — see the next topic for the "why."
Beginner mistakeEyeballing spacing instead of using Figma's inspect panel, leading to slightly-off, inconsistent layouts.
Intermediate mistakeHardcoding raw hex colors copied from Figma throughout the codebase instead of naming them once as design tokens.

Style & StyleSheet

You saw StyleSheet.create briefly in Section 1. Let's go deeper on how styling actually works, and the full set of common style properties you'll use daily.

Three ways to apply styles

styling-approaches.tsx 
// 1. Inline — fine for one-off, dynamic values
<View style={{ marginTop: 20 }} />

// 2. StyleSheet.create — the standard for static, reusable styles
const styles = StyleSheet.create({ box: { padding: 16 } });
<View style={styles.box} />

// 3. Array of styles — merge static + conditional + inline together
<View style={[styles.box, isActive && styles.active, { opacity: fade }]} />
Rule of thumb Use StyleSheet.create for anything static and reused. Reserve inline styles for values that are genuinely dynamic and computed at render time (like an animated value). Mixing both via the array syntax is completely normal and expected.

Properties you'll use constantly

CategoryProperties
Spacingpadding, margin, gap (+ Top/Bottom/Left/Right/Horizontal/Vertical variants)
Sizewidth, height, minWidth, maxHeight, aspectRatio
Layoutflex, flexDirection, justifyContent, alignItems, gap
AppearancebackgroundColor, borderRadius, borderWidth, borderColor, opacity
TextfontSize, fontWeight, color, lineHeight, textAlign
Positionposition, top/left/right/bottom, zIndex
Beginner mistakeWriting sizes with units, like '16px' — React Native style values are unitless numbers, treated as density-independent pixels.
Intermediate mistakeUsing margin on a child to create spacing between siblings instead of gap on the parent — gap is cleaner and avoids margin-collapsing confusion.
Professional mistakeNot memoizing large computed style arrays that depend on rapidly-changing values, causing unnecessary re-renders in performance-sensitive lists.

Quick check

How should fontSize: 16 be written if you wanted it to equal "16px" from CSS?

Correct — React Native uses unitless numbers for nearly all size-related style values.

Design System Tips

A design system is a small set of shared constants and components that every screen pulls from, instead of each screen inventing its own colors, spacing, and font sizes.

theme/tokens.ts 
export const colors = {
  primary: '#ff5a3c',
  text: '#1a1d24',
  textMuted: '#5c6070',
  background: '#fbf9f4',
  border: '#e4ded1',
};

export const spacing = { xs: 4, sm: 8, md: 16, lg: 24, xl: 32 };

export const radius = { sm: 6, md: 12, lg: 20 };

export const typography = {
  heading: { fontSize: 24, fontWeight: '700' },
  body: { fontSize: 15, fontWeight: '400' },
};
  • Naming spacing sm/md/lg instead of raw numbers everywhere means a single change to spacing.md adjusts every screen using it consistently.
  • These tokens should mirror the named styles your designer set up in Figma — same names, same values, both sides of the design/dev handoff stay in sync.

Using tokens in a component

components/Card.tsx 
import { colors, spacing, radius } from '../theme/tokens';

const styles = StyleSheet.create({
  card: {
    backgroundColor: colors.background,
    padding: spacing.md,
    borderRadius: radius.md,
    borderWidth: 1,
    borderColor: colors.border,
  },
});
Why this matters at scale On a team of five engineers, without shared tokens you'll end up with a dozen near-identical "grays" and inconsistent spacing across screens. A tokens file turns a subjective design decision into a single source of truth everyone imports.
Beginner mistakeDefining colors/spacing separately in every component file instead of one shared source.
Intermediate mistakeCreating tokens but still hardcoding raw values "just this once" — inconsistencies creep back in immediately.
Professional mistakeOver-engineering a token system with excessive abstraction layers before the team or app is large enough to need it.
  • Beginner Create a theme/tokens.ts file and refactor one existing component to use it instead of raw values.

TOPIC 4 / 5

Layout with Flexbox

Flexbox is the only layout system in React Native — there's no Grid, no floats, no positioning tricks from web CSS. Mastering it is non-negotiable, and the good news is React Native's default flex direction actually makes some things simpler than on the web.

Key difference from web CSS On the web, the Flexbox default flex-direction is row. In React Native, the default is column — because mobile screens are naturally vertical, and most layouts stack top-to-bottom by default.

The three properties that control almost everything

?
flexDirection

'column' (default) stacks children vertically. 'row' lays them out horizontally.

?
justifyContent

Aligns children along the main axis (matches flexDirection). Values: flex-start, center, flex-end, space-between, space-around.

?
alignItems

Aligns children along the cross axis (perpendicular to flexDirection). Same value options as above, plus stretch.

Flexbox playground

flex: 1 — the property that confuses everyone at first

flex-example.tsx 
<View style={{ flex: 1 }}>
  <View style={{ flex: 1, backgroundColor: 'red' }} />
  <View style={{ flex: 2, backgroundColor: 'blue' }} />
</View>

flex: 1 on the outer View means "take up all available space." Inside it, the red box gets 1 share of space and the blue box gets 2 shares — blue ends up twice as tall as red.

Beginner mistakeConfusing justifyContent and alignItems — remember: justifyContent always follows the main axis (the same direction as flexDirection).
Intermediate mistakeGiving every child a fixed height instead of letting flex distribute space, producing layouts that break on different screen sizes.
Professional mistakeNot testing layouts against both a small phone and a tablet-sized screen, where Flexbox assumptions about available space can break down.

Quick check

In React Native, what is the default value of flexDirection?

Correct — this is one of the most common surprises for developers coming from web CSS, where row is the default.
  • Beginner Using the playground above as reference, build a row of 3 equally-sized boxes using flex: 1 on each.
  • Intermediate Build a header layout: a title on the left, an icon on the right, using justifyContent: 'space-between'.

Dark Mode

Supporting dark mode properly means your color tokens change based on the system setting — not sprinkling if (isDark) checks through every component.

hooks/useThemeColors.ts 
import { useColorScheme } from 'react-native';

const lightColors = { background: '#fbf9f4', text: '#1a1d24' };
const darkColors = { background: '#0f1115', text: '#eceaf0' };

export function useThemeColors() {
  const scheme = useColorScheme(); // 'light' | 'dark' | null
  return scheme === 'dark' ? darkColors : lightColors;
}
  • useColorScheme() — a built-in hook that reads the device's current appearance setting, and automatically re-renders your component if the user switches it while the app is open.
  • By centralizing this in one hook, every component asks "what are my colors right now?" without needing to know how dark mode is detected.

Using it in a component

app/index.tsx 
export default function Home() {
  const colors = useThemeColors();
  return (
    <View style={{ flex: 1, backgroundColor: colors.background }}>
      <Text style={{ color: colors.text }}>Hello</Text>
    </View>
  );
}
Hello

Try it

This simulates useColorScheme() flipping and the same component instantly re-rendering with new colors — no manual theme-switching logic needed per component.

Don't forget these too Status bar style (StatusBar component's barStyle), image assets that need dark variants, and any hardcoded colors buried inside third-party component props are common things people forget when adding dark mode after the fact.
Beginner mistakeHardcoding backgroundColor: '#fff' directly on screens instead of pulling from theme colors — this breaks the moment dark mode is added.
Intermediate mistakeAdding dark mode late in a project, after dozens of components already hardcode light-mode colors — always design the token system with both modes in mind from the start.
Professional mistakeNot testing color contrast in dark mode specifically — colors that look fine in light mode can fail accessibility contrast requirements when inverted.

Section 4 final check

What does useColorScheme() return?

Correct — it returns 'light', 'dark', or null, and updates automatically if the user changes their system setting.
